        # **`Referential Equality`는 무엇인가요?**
        
        간단히 말하면, 
        
        > **참조 동일성**은 “두 변수가 **완전히 같은 객체를 가리키고 있는가**?”를 보는 개념이에요.
        > 
        - “같은 값이냐?”가 아니라 “같은 **메모리 주소(같은 객체 인스턴스)**를 가리키냐?”를 묻는 거에요.
        
        그래서 **“두 객체가 진짜로 같은 녀석이냐?”** 를 확인할 때 쓰는 기준이라고 볼 수 있어요.
        
        ---
        
        ## 1. 자바스크립트에서 값 vs 참조에요
        
        ---
        
        ### 1-1. 값(primitive) 타입
        
        - `number`, `string`, `boolean`, `null`, `undefined`, `bigint`, `symbol`
        - 변수 안에 **값 자체가 들어있는 경우**에요.
        
        ```jsx
        let a = 10;
        let b = 10;
        
        console.log(a === b); // true (값 비교)
        ```
        
        ---
        
        ### 1-2. 참조(object) 타입
        
        - `object`, `array`, `function`, `Date`, `Map` 등
        - 변수 안에는 “객체가 저장된 메모리 위치(참조)”가 들어있어요.
        
        ```jsx
        const obj1 = { x: 1 };
        const obj2 = { x: 1 };
        const obj3 = obj1;
        
        console.log(obj1 === obj2); // false (서로 다른 객체(주소값)에요)
        console.log(obj1 === obj3); // true  (같은 객체(주소값)를 가리켜요)
        ```
        
        - `obj1`과 `obj2`는 모양은 같지만 **서로 다른 객체**에요.
        - `obj1`과 `obj3`는 **정확히 같은 객체**를 공유해요.
            
            → 이 둘이 “참조 동일”하다고 말하는 거예요.
            
        
        ---
        
        ## 2. 자바스크립트에서 ==, ===, Object.is 에요
        
        ---
        
        ### 2-1. `==` (느슨한 동등)
        
        - 타입을 **암묵적으로 변환**하면서 비교해요.
        - 예측하기 어려운 경우가 많아서, **실무에서는 거의 안 쓰는 게 좋아요.**
        
        ```jsx
        0 == false;   // true
        "" == false;  // true
        null == undefined; // true
        ```
        
        ---
        
        ### 2-2 `===` (엄격한 동등)
        
        - 타입 변환 없이 **그대로 비교**해요.
        - primitive에서는 “값 동일성”
        - 객체/배열/함수에서는 “참조 동일성”을 비교해요.
        
        ```jsx
        // primitive
        1 === 1;       // true
        "hi" === "hi"; // true
        
        // object
        const a = { x: 1 };
        const b = { x: 1 };
        const c = a;
        
        a === b; // false (서로 다른 객체)
        a === c; // true  (참조 동일성)
        ```
        
        ---
        
        ### 2-3. `Object.is`
        
        - 기본적으로 `===`와 비슷하지만,
            - `NaN` 비교, `+0` vs `-0` 같은 특수 케이스에서만 조금 달라요.
        
        ```jsx
        NaN === NaN;       // false
        Object.is(NaN, NaN); // true
        
        +0 === -0;        // true
        Object.is(+0, -0);   // false
        ```
        
        - **객체에 대해서는 `===`와 똑같이 참조 동일성**을 비교해요.
        
        ```jsx
        const obj1 = {};
        const obj2 = obj1;
        
        Object.is(obj1, obj2); // true (같은 참조에요)
        ```
        
        ---
        
        ## 3. 객체/배열/함수에서 참조 동일성 예제
        
        ---
        
        ### 3-1. 객체
        
        ```jsx
        const user1 = { name: "kim" };
        const user2 = { name: "kim" };
        const user3 = user1;
        
        console.log(user1 === user2); // false
        console.log(user1 === user3); // true
        ```
        
        - `user1`과 `user2`는 **값은 같아 보여도** 서로 다른 객체에요.
        - `user1`과 `user3`는 같은 참조를 공유하니까 참조 동일이에요.
        
        ---
        
        ### 3-2. 배열
        
        ```jsx
        const arr1 = [1, 2, 3];
        const arr2 = [1, 2, 3];
        const arr3 = arr1;
        
        console.log(arr1 === arr2); // false
        console.log(arr1 === arr3); // true
        ```
        
        “배열 내용이 같은지”를 보려면 **직접 비교 로직**이 필요해요.
        
        ```jsx
        const isArrayEqual = (a, b) => 
          a.length === b.length && a.every((v, i) => v === b[i]);
        
        console.log(isArrayEqual(arr1, arr2)); // true (값 동일성)
        ```
        
        ---
        
        ### 3-3. 함수
        
        ```jsx
        function foo() {}
        
        const bar = function () {};
        
        const baz = foo;
        
        console.log(foo === bar); // false (다른 함수 객체)
        console.log(foo === baz); // true  (같은 함수 객체)
        ```
        
        - 함수도 결국 **객체**라서, 참조 단위로 비교돼요.
        
        ---
        
        ## 4. 참조 동일성이 중요한 이유
        
        ---
        
        ### 4-1. 가변 객체 공유로 인한 사이드 이펙트
        
        ```jsx
        const state = { count: 0 };
        
        function increment(s) {
          s.count += 1;
          return s;
        }
        
        const a = state;
        const b = increment(state);
        
        console.log(a === b);     // true (같은 객체)
        console.log(state.count); // 1
        console.log(a.count);     // 1
        console.log(b.count);     // 1
        ```
        
        - `state`, `a`, `b` 세 변수가 **같은 객체를 공유**하고 있어요.
        - 어느 한 곳에서 `count`를 바꾸면 **전부 동시에 바뀌는 효과**가 나요.
        - “여기만 바꾼 줄 알았는데, 저기도 같이 바뀌었다”는 버그가 나기 쉬운 패턴이에요.
        
        ---
        
        ### 4-2. 불변 객체(immutable) + 참조 동일성
        
        ```jsx
        const state = { count: 0 };
        
        function incrementImmutable(s) {
          return { ...s, count: s.count + 1 };
        }
        
        const a = state;
        const b = incrementImmutable(state);
        
        console.log(a === b); // false (새 객체)
        console.log(a.count); // 0
        console.log(b.count); // 1
        ```
        
        - **원본은 그대로 두고, 새로운 객체를 만들어서 반환**해요.
        - 이 패턴을 쓰면
            - 과거 상태를 안전하게 보관할 수 있고
            - “참조가 바뀌었는지”만 봐도 변경 여부를 어느 정도 알 수 있어요.
        
        리액트에서
        
        - `useMemo`, `useCallback`
        - `React.memo`, `shouldComponentUpdate`
        - 리덕스의 상태 비교
        
        같은 곳에서 **“참조가 바뀌었냐?”를 기준으로 리렌더링/계산 최적화를 하는 이유**가 여기에 있어요.
        
        ---
        
        ## 5. 자바스크립트에서 자주 하는 실수들
        
        ---
        
        ### 5-1. “객체 내용이 같은데 왜 false죠?”
        
        ```jsx
        const a = { x: 1 };
        const b = { x: 1 };
        
        console.log(a === b); // false
        ```
        
        - 이 비교는 **“같은 객체냐?”** 를 묻는 거예요.
        - “내용이 같냐?”를 보려면 별도의 **deepEqual** 로직이나 라이브러리를 써야 해요.
        
        ---
        
        ### 5-2. 배열도 마찬가지
        
        ```jsx
        [1, 2, 3] === [1, 2, 3]; // 항상 false
        ```
        
        매번 새로 만들어지는 배열은 **다 다른 객체**라서, 참조 동일성이 깨져요.
        
        ---
        
        ### 5-3. 상태를 직접 수정하면서, 참조 비교로 변경 여부를 보려는 경우
        
        ```jsx
        const state = { x: 1 };
        
        function mutate(s) {
          s.x = 2;
        }
        
        mutate(state);
        
        console.log(state === state); // 항상 true죠…
        ```
        
        - “객체를 직접 수정”했는데, 참조는 그대로니까 참조 동일성으로는 변경 여부를 알 수 없어요.
        - 참조 비교로 변경 여부를 감지하고 싶다면 **불변 패턴(새 객체 생성)**을 써야 해요.
        
    - **리액트 렌더링 최적화**와 **`Referential Equality`**는 어떤 관계가 있을까요? 🍠
        
        리액트는 객체의 **내용을 비교하지 않고 참조 동일성(`===`)** 으로 props와 state의 변경 여부를 판단.
        
        따라서 객체·배열·함수를 새로 생성하면 값이 같아도 참조가 달라져 리렌더링이 발생!
        
        `React.memo`, `useMemo`, `useCallback`은 이러한 **불필요한 참조 변경을 막아,**  리렌더링과 재계산을 최적화하기 위한 도구.
        
        이를 제대로 활용하려면 상태를 **불변 객체로 관리**해 참조 변경이 의미 있는 경우에만 발생하도록 해야 함.

            # **`useCallabck`** 이 무엇인지?
            
            ---
            
            - 함수(콜백)를 “메모이제이션” 한다는 게 무슨 뜻인지?
                - 함수의 **실행 결과를 저장하는 게 아니라**,
                    
                    **함수 객체 자체(참조)를 저장**하는 것이다.
                    
                - dependencies가 같으면 → **이전 함수 그대로 반환**
                - dependencies가 바뀌면 → **새 함수 생성**
            - 언제 새 함수를 만들고, 언제 기존 함수를 재사용하는지?
                - 컴포넌트가 다시 렌더링되어도 **의존성이 변하지 않으면 같은 함수 참조를 재사용**한다.
                - 즉, 매 렌더마다 새 함수를 만들지 않고 **기존 함수를 유지**해준다.
                - 의존성이 변할 때만 함수를 생성.
        - 왜 **`useCallabck`**을 사용하는지? 🍠
            
            # 왜 **`useCallabck`**을 사용하는지?
            
            ---
            
            - **불필요한 리렌더링 방지**와 어떤 관련이 있는지
                - React는 props를 `===`로 비교한다.
                - 매 렌더마다 새 함수가 생성되면 참조가 달라져
                    
                    `React.memo`로 감싼 자식도 다시 렌더링된다.
                    
                - `useCallback`은 **함수 참조를 고정**해 이를 막아준다.
            - 성능 최적화 관점에서 얻는 이득 vs 남용했을 때의 오버헤드
                - `useCallback`은 함수 참조를 유지해 **불필요한 리렌더링과 재계산을 줄이는 이점**이 있다.
                - 특히 `React.memo`로 감싼 자식 컴포넌트에 콜백을 전달할 때 효과적이다.
                - 하지만 모든 함수에 사용하면 **의존성 비교 비용과 코드 복잡도**가 증가한다.
                - 따라서 실제 성능 이득이 있는 경우에만 **선별적으로 사용하는 것이 중요**하다.
        - **`useCallabck`** 기본 사용법 🍠
            
            # **`useCallabck`** 기본 사용법
            
            ---
            
            - **`useCallabck`**은 어떻게 사용하나요? (코드)
                
                ```tsx
                import { useCallback } from "react";
                
                const handleClick = useCallback(() => {
                  console.log("click");
                }, []);
                ```
                
            - `deps` 배열에 무엇을 넣어야 하는지 규칙
                - 콜백 내부에서 사용하는 **state, props, 변수는 모두 deps에 포함**해야 한다.
                - deps는 **항상 고정된 길이의 배열**이어야 한다.
                - React는 deps를 `Object.is`로 비교한다.
            - 의존성 변경 시 콜백이 어떻게 다시 만들어지는지
                - `useCallback`은 렌더링 시마다 **의존성 배열(deps)을 이전 값과 비교**한다.
                - React는 각 의존성을 `Object.is`로 비교해 변경 여부를 판단한다.
                - deps 중 하나라도 변경되면 **기존 콜백을 버리고 새 함수를 생성**한다.
                - 모든 의존성이 동일하면 **이전 렌더에서 생성된 콜백을 그대로 재사용**한다.
        - **`useCallabck`**에서 중요한 개념 🍠
            
            # **`useCallabck`**에서 중요한 개념
            
            ---
            
            - **참조 동일성(reference equality)** 이 왜 중요한지 (=== 비교)
                - React는 객체·배열·함수의 변경 여부를 **값이 아니라 참조 동일성(`===`)** 으로 판단한다.
                - 참조가 같으면 “변경 없음”, 참조가 다르면 “변경 있음”으로 인식한다.
                - 따라서 값이 같아도 새 객체나 새 함수가 생성되면 **불필요한 리렌더링**이 발생할 수 있다.
                - `useCallback`, `useMemo`, `React.memo`는 **참조를 유지해 성능 최적화**를 가능하게 한다.
            - 클로저와 상태: 콜백 안에서 state, props를 사용할 때 주의할 점
                - 콜백 함수는 **생성 시점의 state와 props를 클로저로 캡처**한다.
                - 따라서 `useCallback`의 deps에 state나 props를 누락하면 **최신 값이 반영되지 않을 수 있다**.
                - 콜백 내부에서 state를 기반으로 업데이트할 경우, **함수형 업데이트(`prev => …`)를 사용하는 것이 안전**하다.
                - 항상 콜백에서 참조하는 값이 **의존성 배열에 정확히 포함되어 있는지** 확인해야 한다.
            - **stale closure(낡은 값 캡처)** 문제는 언제 생기는지, 어떻게 피하는지
                - `useCallback`으로 만든 함수가 **생성 당시의 state/props 값을 클로저로 기억**할 때 발생한다.
                - deps 배열에 필요한 state나 props를 빠뜨리면, **최신 값이 아닌 과거 값**을 계속 사용하게 된다.
                - 이를 피하려면 **콜백에서 사용하는 모든 값들을 deps에 정확히 포함**해야 한다.
                - 상태 업데이트가 목적이라면 `setState(prev => …)` 같은 **함수형 업데이트**를 사용하면 stale closure를 안전하게 피할 수 있다.

            # **`useCallabck`**을 사용한 콜백 메모이제이션 예시
            
            ---
            
            - 부모에서 자식으로 콜백을 내려줄 때, `onClick`, `onChange` 같은 핸들러를 **`useCallabck`** 없이 넘겼을 때와 **`useCallabck`**으로 감싸서 넘겼을 때 차이
                
                ### ❌ `useCallback` 없이 콜백 전달
                
                ```tsx
                functionParent() {
                consthandleClick = () => {
                console.log("click");
                  };
                
                return<ChildonClick={handleClick} />;
                }
                ```
                
                - 렌더링마다 **새 함수가 생성**
                - `Child`가 `memo`로 감싸져 있어도 **항상 리렌더링됨**
                
                ---
                
                ### ✅ `useCallback`으로 감싼 경우
                
                ```tsx
                functionParent() {
                const handleClick =useCallback(() => {
                console.log("click");
                  }, []);
                
                return<ChildonClick={handleClick} />;
                }
                ```
                
                - 함수 참조가 유지됨
                - `Child`는 props가 같다고 판단해 **리렌더링을 건너뜀**
                
                ---
                
                ### 정리 한 줄 ✏️
                
                > useCallback은 부모 렌더링 시에도 콜백 참조를 유지해
                자식 컴포넌트의 불필요한 리렌더링을 막는다.

            # **`memo`**가 무엇인지?
            
            ---
            
            - `memo`는 **props가 변경되지 않으면 컴포넌트 리렌더링을 건너뛰는** 고차 컴포넌트(HOC)이다.
            - 부모 컴포넌트가 리렌더링되어도, **props가 이전과 같으면 자식은 리렌더링되지 않는다**.
            - 기본적으로 React는 props를 `Object.is`(얕은 비교)로 비교한다.
            - 단, `memo`는 **성능 최적화 도구이지 렌더링 보장을 의미하지는 않는다**.
            
        - 왜 **`memo`**를 사용하는지? 🍠
            
            # 왜 **`memo`**를 사용하는지?
            
            ---
            
            - 렌더링 비용이 큰 컴포넌트의 **불필요한 리렌더링을 방지**하기 위해 사용한다.
            - 부모 상태 변경이 잦지만, 자식 props는 거의 변하지 않는 경우에 효과적이다.
            - `useCallback`, `useMemo`와 함께 사용하면 **참조 동일성 유지로 최적화 효과가 커진다**.
            - 결과적으로 **UI 반응성과 성능을 개선**할 수 있다.
        - **`memo`** 기본 사용법 🍠
            
            # **`memo`** 기본 사용법
            
            ---
            
            ```tsx
            import { memo }from"react";
            
            constChild =memo(functionChild({ value }) {
            return<div>{value}</div>;
            });
            ```
            
            - `memo(Component)` 형태로 감싸서 사용한다.
            - props가 이전과 동일하면 컴포넌트 렌더링을 건너뛴다.
            - 필요하면 두 번째 인자로 **커스텀 비교 함수**를 전달할 수 있다.

            # **`memo`**를 언제 쓰면 좋은지 / 안 좋은지
            
            ---
            
            ### ✅ 쓰기 좋은 경우
            
            - 렌더링 비용이 크고 props 변경이 드문 컴포넌트
            - 리스트 아이템, 차트, 복잡한 UI 컴포넌트
            - props로 객체·함수 전달 시 `useCallback`/`useMemo`와 함께 사용
            
            ### ❌ 안 좋은 경우
            
            - 렌더링 비용이 매우 작은 컴포넌트
            - props가 자주 바뀌는 컴포넌트
            - 무분별하게 사용하면 **비교 비용과 코드 복잡도만 증가**

        # **`배포`**는 무엇인가요?
        
        ---
        
        **`배포`(Deployment)**는 개발자가 만든 코드를 실제 사용자들이 사용할 수 있도록 인터넷 상에 공개하는 과정이에요. 내 컴퓨터(Localhost)에서만 돌아가던 웹사이트를, 누구나 URL을 통해 접속할 수 있는 (Server)로 옮기는 작업이라고 생각하면 좋아요.
        
        ---
        
        **특징 정리:**
        
        - **빌드(Build):** 우리가 작성한 코드(React, TypeScript 등)를 브라우저가 이해할 수 있는 파일(HTML, CSS, JS)로 변환하는 과정이 필요해요.
        - **호스팅(Hosting):** 변환된 파일들을 24시간 켜져 있는 컴퓨터(서버)에 저장해 두고, 요청이 오면 보여줘요.
        - **접근성:** 배포가 완료되면 고유한 도메인 주소(예: `www.yolog.co.kr`)가 생겨서 어디서든 접속할 수 있어요.
        
        ---
        
        **단점도 있어요:**
        
        - 과거에는 직접 서버 컴퓨터를 사고, 리눅스를 설치하고, 복잡한 네트워크 설정을 해야 했어요.
        - 코드를 수정할 때마다 수동으로 파일을 서버에 업로드해야 해서 번거롭고 실수가 잦았어요.

        # **Vercel**은 무엇인가요?
        
        ---
        
        **Vercel**은 프론트엔드 개발자가 복잡한 서버 설정 없이 아주 쉽고 빠르게 웹사이트를 배포할 수 있도록 도와주는 플랫폼이에요.
        React의 프레임워크인 **Next.js**를 만든 팀에서 개발했기 때문에, 리액트 기반 프로젝트와 찰떡궁합인 도구라고 생각하면 좋아요.
        
        ---
        
        **특징:**
        
        - **GitHub 연동:** GitHub 저장소에 코드를 올리기만 하면(Push), Vercel이 알아서 감지하고 배포를 시작해요.
        - **자동 설정:** 복잡한 서버 세팅 없이도 HTTPS(보안 연결), 도메인 설정 등을 자동으로 해줘요.
        - **미리보기(Preview):** 실제 서비스에 반영하기 전에, 테스트 주소를 생성해 줘서 팀원들과 미리 확인해 볼 수 있어요.
        - **빠른 속도:** 전 세계 곳곳에 서버(CDN)가 있어서 사용자가 어디에 있든 빠르게 사이트를 보여줘요.
        
    - **CI / CD**는 무엇인가요?
        
        # **CI / CD**는 무엇인가요?
        
        ---
        
        **CI/CD**는 개발부터 배포까지의 과정을 "자동화"하여, 더 자주, 더 안전하게 코드를 배포할 수 있게 하는 방법론이자 도구 모음이에요.
        개발자가 일일이 수동으로 하던 일들을 "로봇(스크립트)"에게 맡겨서 실수를 줄이는 것이 핵심이에요.
        
        ---
        
        ### **1. CI (Continuous Integration - 지속적 통합)**
        
        - **무엇인가요?** 여러 개발자가 작성한 코드를 **자주, 정기적으로 합치는(Merge)** 과정이에요.
        - **하는 일:** 코드가 합쳐질 때마다 자동으로 **테스트**를 실행하고, 빌드가 잘 되는지 확인해요.
        - **왜 하나요?** 버그를 미리 발견해서, 잘못된 코드가 합쳐지는 것을 막기 위해서예요. "문지기" 역할을 한다고 보면 돼요.
        
        ---
        
        ### **2. CD (Continuous Deployment/Delivery - 지속적 배포/제공)**
        
        - **무엇인가요?** CI를 통과한 코드를 자동으로 **사용자에게 배포**하는 과정이에요.
        - **하는 일:** 테스트가 끝난 코드를 서버로 전송하고, 실제 웹사이트를 업데이트해요.
        - **왜 하나요?** 개발자가 수동으로 업로드할 필요 없이, 코드만 작성하면 고객에게 기능이 즉시 전달되게 하기 위해서예요.
        
    - **`배포`** vs **Vercel** vs **CI/CD** 한눈에 비교
        
        # 배포 vs Vercel vs CI/CD 한눈에 비교
        
        ---
        
        | **항목** | **배포 (Deployment)** | **Vercel** | **CI / CD** |
        | --- | --- | --- | --- |
        | **핵심 개념** | 서비스 공개 | 배포 도구 (플랫폼) | 자동화 프로세스 |
        | **역할** | 내 컴퓨터 -> 세상 밖으로 | 배포를 쉽게 해주는 조력자 | 개발~배포 과정을 연결하는 파이프라인 |
        | **비유** | 가게 오픈 | 인테리어/설비 다 된 매장 임대 | 로봇이 청소/요리/서빙까지 자동화 |
        | **난이도** | (과거) 매우 높음 | 매우 낮음 (클릭 몇 번) | 설정에 따라 다름 (GitHub Actions 등) |
        | **주요 작업** | 파일 업로드, 서버 실행 | GitHub 레포지토리 연결 | 테스트 코드 작성, 워크플로우(.yml) 설정 |
    - **Vercel과 CI/CD**가 권장되는 이유
        
        # **Vercel과 CI/CD**가 권장되는 이유
        
        ---
        
        현대 개발 환경에서 핵심은 **"생산성"**과 **"안정성"**이에요:
        
        > "개발자는 비즈니스 로직(기능 구현)에만 집중하세요. 배포와 운영은 도구가 해결해 드립니다."
        > 
        
        즉, **Vercel**을 사용하면 인프라 구축 시간을 아낄 수 있고, **CI/CD**를 구축하면 사람이 할 수 있는 실수를 컴퓨터가 대신 검증해주기 때문에 훨씬 안정적인 서비스 운영이 가능해요.
        
        ---
        
        ### **✔ 언제 이 도구들을 사용할까요?**
        
        ---
        
        👉 **이런 경우라면 강력 추천해요**
        
        - ✔ 내가 만든 프로젝트를 포트폴리오로 제출해서 링크로 보여주고 싶을 때 (**Vercel**)
        - ✔ 코드를 수정할 때마다 일일이 다시 빌드하고 업로드하기 귀찮을 때 (**CI/CD**)
        - ✔ 팀 프로젝트를 진행하며, 팀원의 코드가 에러 없이 잘 합쳐지는지 확인하고 싶을 때
        - ✔ 실무에서처럼 체계적인 개발 프로세스를 경험해보고 싶을 때
        
        ---
        
        👉 **이런 경우 과할 수도 있어요**
        
        - ✔ 인터넷 연결 없이 로컬에서만 동작하는 프로그램을 만들 때
        - ✔ 배포할 필요 없이 코드 로직만 공부하는 단계일 때
    - **추가로 본인이 학습한 내용에 대해서 정리해주세요** 🍠
        
        # **추가로 본인이 학습한 내용에 대해서 정리해주세요** 🍠
        
        ---
        
        - **Preview 배포의 중요성**: Vercel은 PR마다 Preview URL을 자동 생성해, 실제 배포 전 기능·UI를 안전하게 검증할 수 있다.
        - **환경 변수 관리**: API Key 같은 민감한 값은 코드에 직접 넣지 않고, Vercel의 Environment Variables로 관리해야 한다.
        - **Rollback의 필요성**: 배포 후 문제가 생기면 이전 버전으로 즉시 되돌릴 수 있어 서비스 안정성이 높아진다.
        - **자동화의 신뢰성**: CI 단계에서 테스트가 실패하면 배포 자체가 차단되어, 오류 코드가 사용자에게 노출되는 것을 막는다.
        - **협업 효율 향상**: GitHub Actions + Vercel을 사용하면 팀원 간 “내 로컬에서는 됐어” 문제를 크게 줄일 수 있다.
        - **실무와의 연결성**: 대부분의 실무 프론트엔드 프로젝트는 CI/CD 기반 배포를 사용하므로, 학습 자체가 실무 경험이 된다.
        - **배포는 개발의 끝이 아니라 시작**: 배포 이후 로그, 성능, 사용자 반응을 보며 지속적으로 개선하는 과정이 중요하다는 것을 이해했다.

**1. 준비 단계 (GitHub & CLI)**

내 코드를 배포할 준비를 하는 단계예요.

- [ ]  **GitHub 레포지토리 생성 및 코드 업로드 (Push)**
    - `git init` → `git add .` → `git commit` → `git push`
    - ⚠️ **주의:** API 키가 들어있는 `.env` 파일은 절대 올리면 안 돼요! (`.gitignore` 확인)
- [ ]  **Vercel CLI 설치**
    - 터미널에 `npm install -g vercel` 입력
- [ ]  **Vercel 로그인**
    - 터미널에 `vercel login` 입력 후 GitHub 계정 등으로 로그인 완료하기

**2. 프로젝트 연결 및 첫 배포 (Preview)**

Vercel과 내 프로젝트를 연결하고 미리보기(Preview)를 띄우는 단계예요.

- [ ]  **Vercel 연동 시작**
    - 프로젝트 경로에서 `vercel` 명령어 입력
- [ ]  **초기 설정 질문 답변하기**
- [ ]  **Preview 배포 확인**
    - 터미널에 뜬 `Preview` URL로 접속해서 사이트가 나오는지 확인하기

**3. 실서비스 배포 (Production)**

이제 진짜 사용자가 볼 수 있는 주소로 배포하는 단계예요.

- [ ]  **Production 배포 명령어 실행**
    - 터미널에 `vercel --prod` 입력
- [ ]  **Production URL 확인**
    - `Production` 옆에 뜬 URL로 접속해서 확인하기

**4. 필수 설정 및 에러 해결**

블로그에서 강조한 **"새로고침 시 404 에러"**와 **"환경변수"** 해결 단계에요.

    - [ ]  제대로 **`/movies/:movieId`** 페이지 이동이 되는지 확인해주세요. (실제로 디자인을 할 필요는 없습니다.)
- [ ]  **환경변수(Environment Variables) 등록**
    - Vercel 대시보드 → Settings → Environment Variables 탭 이동
    - `.env` 파일에 있던 내용(API URL 등)을 똑같이 입력하고 Save

# 🥕 Vercel은 어떻게 무료로 배포를 가능하게 할까?

---

<aside>
🥕

**Vercel**은 AWS CloudFront처럼 트래픽 비용을 직접 지불해야 하는 방식과 달리, 개인 개발자에게 **완전 무료 배포**를 제공하며 프론트엔드 배포의 **치트키**로 자리 잡았어요.

사실 Vercel은 자체 서버가 아닌 **AWS 인프라** 위에서 구동되지만, 복잡한 설정을 걷어내고 **개발자 경험(DX)**을 극대화하는 전략으로 사용자를 모으고 있어요.

개인에게는 비용을 대신 내주며 생태계에 **락인(Lock-in)** 시키고, **기업 및 팀 단위**로 확장될 때 **협업 기능과 보안** 비용을 청구해 수익을 내는 **프리미엄(Freemium) 모델**이 핵심이에요.

즉, AWS가 저렴한 도매상이라면 Vercel은 **편리함과 시간을 제공**하는 대신 성공한 서비스에게 비용을 받는 똑똑한 구조인 셈이죠.

이러한 프론트엔드 배포의 흐름을 한번 재미있게 정리해보았어요.

참고로, 이 글의 내용은 전부 **Perplexity**를 기반으로 정리했습니다.

</aside>

- 📢 학습 회고 (예시, 서식만 복사하시고 지워주세요.)
    - **프론트엔드 배포, Vercel 활용**
        - **이해한 점**: Vercel은 프론트엔드 프로젝트를 **빠르고 간편하게 배포**할 수 있는 플랫폼입니다.
            
            GitHub 연동, 환경변수 설정, 커스텀 도메인 연결 등 배포 과정 대부분이 GUI와 CLI로 쉽게 처리되며, SPA 환경에서도 라우팅 문제를 `vercel.json` 설정으로 해결할 수 있습니다.
            
            - 예시:
                
                ```bash
                # CLI로 배포
                vercel         # Preview 배포
                vercel --prod  # Production 배포
                ```
                
                ```json
                // SPA 라우팅 문제 해결
                {
                  "routes": [
                    { "src": "/[^.]+", "dest": "/index.html", "status": 200 }
                  ]
                }
                ```
                
        - **어려운 점 (개선 방법)**: SPA 기반 프로젝트는 새로고침 시 404 문제가 발생할 수 있으며, 환경변수 관리, 커스텀 도메인 연결 과정이 처음에는 헷갈렸습니다.
            - 개선 방법: `vercel.json` 설정으로 SPA 라우팅 문제를 해결하고, Vercel Dashboard에서 환경변수와 DNS 설정을 직접 확인하면서 반복적으로 배포 실습을 진행했습니다.
            - 예시:
                
                ```tsx
                // 환경변수 사용
                const api = axios.create({
                  baseURL: import.meta.env.VITE_API_URL,
                  headers: { 'Content-Type': 'application/json' },
                });
                
                ```
                
        - **회고**: 실제 배포 과정을 경험해보니, 로컬 환경과 다른 실제 서비스 환경에서의 테스트 필요성을 이해할 수 있었습니다.
            
            앞으로 프로젝트를 진행할 때, GitHub 연동과 Vercel 배포를 활용해 **즉시 테스트 가능한 환경**을 만들고, SPA 라우팅 문제와 환경변수를 신경 써서 안정적으로 서비스를 운영할 수 있을 것 같습니다.
