📚 블로그 읽고 Content Security Policy(CSP) 정리해보기 🍠

- 서버에서 HTML 문서를 응답할 때 CSP를 적용하려면 어떤 HTTP 응답 헤더를 설정해야 하나요? 블로그에 나온 Express.js 코드 예시를 기반으로 설명해보세요.
      서버에서 웹 문서를 응답할 때 이 `Content-Security-Policy` 헤더에 적절한 정책을 명시하면 브라우저가 이를 따르게 된다.
      ```jsx
  res.setHeader('Content-Security-Policy',"default-src 'self'");

````

- `default-src 'self'` 설정은 브라우저에게 어떤 보안 정책을 의미하나요? 또한 `'self'` 값은 어떤 출처를 포함하거나 제외하나요?

    `default-src` 는 **fallback**으로 동작한다.

    다른 설정이 없으면 `default-src`의 설정값을 따르며, `‘self’`는 **현재 문서의 출처**를 의미한다.

    위 설정은 브라우저가 **현재 출처의 자원만 사용하라는 기본값**을 명시한 것이다.

    브라우저는 HTML 요청을 하고 응답을 받을 때 `Content-Security-Policy` 응답 헤더를 확인한다.

    `defult-src ‘self’`로 지정하면 이 문서의 출처가 아닌 곳에서 온 리소스는 실행하지 않는다.

- 블로그에 나온 악성 스크립트(`<script>fetch(...)</script>`)를 주입했을 때 CSP가 어떻게 동작하는지 네트워크 탭과 콘솔 메시지 측면에서 설명해보세요.

    브라우저 콘솔에는 Content Security Policy 경고가 표시된다.

    “다음 지시자를 위반했기 때문에 인라인 스크립트가 실행되지 않도록 차단했다”라는 메시지를 볼 수 있다.

    (= 건물 출입구의 보안 요원이 출입증이 없는 사람을 막는 것과 같다. 허가되지 않은 스크립트는 애초에 실행되지 않는다.)

- 기본 CSP 설정에서 인라인 스타일이 차단된다고 했습니다. 블로그 예시 중 `width:600px`이 적용되지 않는 이유를 설명하세요.

    **인라인 스타일이 차단되는 이유 (보안 취약점)**

    CSP의 기본적인 설정은 **가장 안전한 것만 허용**하는 것을 목표로 한다. 이 목표에 따라, 기본적으로 **인라인 스크립트**와 **인라인 스타일**은 위험하다고 판단되어 차단된다.

    **왜 인라인 스타일이 위험할까**

    악성코드를 심으려는 해커가 인라인 스타일을 이용할 수 있기 때문이다.

    - 예시: 해커가 사용자의 비밀번호 입력란을 투명하게 만들거나(`style=”opacity:0;”`), 화면 바깥으로 숨겨서 (`style=”display:none;”`) 사용자가 눈치채지 못하게 악성 행위를 유도할 수 있다.
    - CSP가 인라인 스타일 자체를 차단해버리면, 해커가 태그에 스타일을 몰래 심으려고 해도 “차단”당하게 되어 웹사이트 보안이 훨씬 강화된다.

    **CSP 설정의 작동 방식**

    - `script-src ‘self’`: 스크립트는 **같은 출처**에 있는 파일만 허용. 인라인 스크립트 차단.
    - `style-src ‘self’`: 스타일은 **같은 출처**에 있는 파일만 허용. 인라인 스타일 차단.

    **적용이 안되는 이유**

    1. 적용하려는 코드 : <div style=”width:600px;”>와 같이 HTML 태그 안에 직접 넣은 인라인 스타일이다.
    2. CSP의 규칙: 현재 설정된 CSP는 보안 강화를 위해 **모든 인라인 스타일을 잠재적으로 위협으로 간주하고 차단**하도록 설정되어 있다.
    3. 결과: CSP 규칙에 따라 브라우저는 해당 인라인 스타일을 무시하고 적용하지 않기 때문에 문서에는 width:600px이 반영되지 않는 것.

- 구글 애널리틱스, 카카오맵, 외부 API 등이 CSP 때문에 차단될 수 있다고 했습니다. 이러한 현상을 "건물 보안을 강화한다"는 비유와 연결해 설명해보세요.

    CSP를 너무 엄격하게 설정하면 정상적인 기능까지 제한될 수 있다.

    **실제 발생할 수 있는 문제들:**

    - **구글 애널리틱스** 같은 외부 분석 도구가 작동하지 않음
    - **카카오맵, 네이버맵** 등 일부 API가 차단됨
    - **소셜 로그인** 버튼이 작동하지 않음
    - **웹폰트**가 로드되지 않아 디자인이 깨짐

    이는 마치 건물 보안을 강화한다고 모든 출입문을 막아버려서, 정작 출입해야하는 직원들도 출입하지 못하는 상황과 같다.

    따라서 CSP는 **단계적으로 적용**하면서 서비스에 미치는 영향을 **지속적으로 모니터링**해야한다.


- Report-Only 모드에서는 실제 리소스 실행이 차단되지 않습니다. 그 대신 브라우저와 서버에서 각각 어떤 동작을 수행하나요?

    Report-Only 모드는 웹사이트를 정상 작동시키면서, 동시에 개발자에게 **만약 보안 규칙을 적용하면 여기서 문제가 생깁니다**라고 알려주는 **무료 감시 시스템** 역할을 한다.

    CSP를 단계적으로 적용하기 위한 방법을 Report-Only 모드가 있다. 이 모드에서는:

    1. 브라우저가 정책 위반을 **진단만** 하고
    2. 리소스 로드는 **차단하지 않고** 그대로 실행
    3. 진단 결과를 **서버로 보고**

    이를 통해 실제 서비스에 영향을 주지 않으면서 잠재적인 보안 위협을 파악할 수 있다.

    마치 건물에 CCTV를 먼저 설치해서 출입 패턴을 분석한 다음, 그 데이터를 바탕으로 보안 규칙을 만드는 것과 같다.

    **브라우저가 수행하는 동작**

    1. 리소스 실행 (차단하지 않음)

        CSP 규칙을 위반하는 콘텐츠 (예: 인라인 스타일, 외부 스크립트)를 차단하지 않고 정상적으로 실행한다.

        →사용자 경험에 영향을 주지 않고, 웹사이트가 깨지는 것을 방지

    2. 보고서 생성 및 전송

        CSP 규칙 위반이 발생하면, 브라우저는 해당 위반에 대한 상세정보를 담은 JSON 형식의 보고서를 즉시 생성한다

        - 보고서 내용: 어떤 규칙을 위반했는지, 위반한 리소스의 주소는 무엇인지, 현재 페이지의 주소는 무엇인지 등 자세한 정보가 포함된다
        - 전송 방식: 이 보고서를 CSP 헤더에 저장된 report-uri 또는 report-to 주소로 HTTP POST 요청을 이용해 서버로 전송한다.

    **서버가 수행하는 동작**

    1. 보고서 수신 및 저장

        서버는 브라우저가 보낸 위반 보고서(JSON 데이터))를 수신하는 전용 엔드포인트(URL)를 운영해야 한다.

        동작: 서버는 이 보고서를 받아서 로그 파일, 데이터베이스 등에 저장한다.

    2. 데이터 분석 (잠재적 문제 파악)

        저장된 보고서 데이터를 분석하여 웹사이트의 잠재적인 문제점을 파악하고 수정한다.

        - 목표: 어떤 인라인 스크립트/스타일이 가장 많이 차단되는지, 어떤 외부 도메인 리소스가 필요한지 등을 파악한다.
        - 예시: “로그를 분석해보니, 우리가 만든 결제 모듈에서 `style-src`규칙을 100번 위반했네. 이 모듈을 수정하거나 CSP에 해당 스타일을 허용하도록 예외를 추가해야겠다.”
        - 결과: 실제 CSP를 적용했을 때 발생할 수 있는 오류나 기능 마비를 미리 예측하고 방지할 수 있다.

- CSP만으로는 CSRF를 막을 수 없다고 했습니다. 블로그에 정리된 다른 보안 조치들(SameSite 쿠키, X-Frame-Options 등) 중 2가지를 설명하세요.

    ### **SameSite 쿠키 (CSRF 공격 방어의 핵심)**

    기본적인 개념: 쿠키의 ‘출처’ 제한

    SameSite는 브라우저가 **쿠키를 요청에 첨부할지** 결정하는 규칙이다. 특히, 사용자가 현재 접속해 있는 사이트와 **다른 출처(다른 사이트)**에서 요청이 왔을 때, 민감한 쿠키(ex 세션쿠키)가 자동으로 전송되는 것을 막아 **CSRF 공격을 방어**하는 데 핵심적인 역할을 한다.

    **왜 필요한지? (CSRF 공격 시나리오)**

    CSRF 공격은 사용자가 로그인한 상태에서, 해커의 웹사이트에 접속했을 때 해커가 사용자 몰래 피해 사이트(예: 은행 사이트)로 요청을 보내는 공격.

    이때, 브라우저는 보통 이 요청에 **사용자의 세션 쿠키**를 자동으로 첨부한다. 피해 사이트는 이 쿠키를 보고 **정상적인 사용자의 요청**이라고 착각하고 실행하게 된다.

    **SameSite 속성의 동작방식**

    | 속성 | 동작원리 | 보안효과 |
    | --- | --- | --- |
    | Lax | 가장 보편적인 설정. 사용자가 **링크를 클릭**하거나 <form>으로 **GET 요청**을 보낼 때처럼, ‘사용자의 의도’가 명확한 **최상위 탐색**을 위한 요청에만 쿠키를 첨부한다. 다른 사이트에서 악의적인 스크립트나 이미지 로드를 통해 요청을 보낼 때는 **쿠키를 첨부하지 않아** CSRF 공격을 방어한다. | CSRF 방어에 매우 효과적이며, 사용성도 좋다. |
    | Strict | 가장 강력한 설정. 같은 사이트(출처, 도메인)에서 온 요청에만 쿠키를 첨부한다. **다른 사이트에서 링크를 클릭**하여 들어오는 요청(새로운 탐색)에도 쿠키가 전혀 첨부되지 않아 보안은 강력하지만, 사용자가 불편함을 느낄 수 있다. | 강력한 CSRF 방어 |

    ⇒결론: SameSite 쿠키를 Lax로 설정하면, 해커의 사이트에서 보내는 위조된 요청에는 세션 쿠키가 포함되지 않으므로, 서버가 그 요청을 무시하게 되어 CSRF 공격이 실패한다.

# **📚 블로그 읽고 동일 출처 정책(Same Origin Policy) 정리해보기 🍠**

**CSP(Content Security Policy)**가 서버에서 정책을 설정하고 브라우저가 수행하는 보안 시스템이라면, **SOP(Same Origin Policy)**는 **브라우저가 스스로 자원의 출처를 관리하는 정책**이다.

목적: 악의적인 웹사이트가 사용자 데이터를 무단으로 접근하거나 유출하는 것을 방지

예를 들어 Session Hijacking(세션 하이재킹)처럼 사용자가 로그인한 세션 쿠키를 다른 사이트로 보내지 못하게 차단한다.

마치 아파트 출입구에서 단지 주민증을 확인하는 것과 같다. 같은 단지 주민만 들어갈 수 있고, 다른 곳에서 온 사람은 기본적으로 차단되는 것

**SOP의 기본원칙**

- **같은 출처의 자원만 접근 가능**: 웹 문서와 같은 출처의 자원만 로드할 수 있음
- **다른 출처의 자원은 기본적으로 차단**: 교차 출처 자원은 특별한 허가 없이 접근 불가

- 출처(Origin)는 어떤 세 요소의 조합으로 결정되나요?
    - **프로토콜** (`http`,`https`)
    - **호스트**(도메인)
    - **포트**(`80`, `443`, `8080`)

    이 세가지가 모두 일치해야 **같은 출처**로 간주된다.

- 출처의 요소가 다른 경우(예: 프로토콜만, 포트만 다른 경우)에 같은 출처인지 아닌지를 예시 3개(같은 출처 1개, 다른 출처 2개)로 설명하세요.

    기준 URL: `https://www.api.com:443/data/user`

    **같은 출처(Same-Origin) 예시**

    - 비교 URL: `https://www.api.com:443/products/list`
    - 프로토콜: `https` (일치)
    - 호스트: `www.api.com` (일치)
    - 포트: `:443` (일치)

    ⇒같은 출처

    URL 경로(/data/user 와 /products/list)가 다르더라도, 프로토콜, 호스트, 포트 세 가지 핵심 요소가 모두 일치하기 때문에 같은 출처로 간주된다. 이 두 페이지/자원은 서로 제한 없이 상호작용(예: 데이터 요청)할 수 있다.

    **다른 출처(Different-Origin)예시: 프로토콜만 다른 경우**

    - 비교 URL: `http://www.api.com:443/data/user`
    - 프로토콜: `http` (다름)
    - 호스트: `www.api.com` (일치)
    - 포트: `:443` (일치)

    ⇒ 다른 출처

    http에서 https 자원에 접근하는 것은 보안 정책(특히 동일 출처 정책, SOP)에 의해 기본적으로 제한된다.

    **다른 출처(Different-Origin)예시: 포트만 다른 경우**

    - 비교 URL: `http://www.api.com:8080/data/user`
    - 프로토콜: `http` (일치)
    - 호스트: `www.api.com` (일치)
    - 포트: `:8080` (다름)

    ⇒ 다른 출처

    웹 서버의 포트는 독립적인 엔드포인트(접속 지점)로 취급되므로, 포트 번호가 다르면 완전히 다른 자원으로 간주되어 상호작용에 제한이 생긴다.

- 블로그에 나온 `fetch` 기반 악성 스크립트를 다른 출처로 실행했을 때 브라우저에서 어떤 일이 발생하나요? 네트워크 전송 여부, 응답 사용 가능성, 브라우저 콘솔 메시지 측면에서 서술하세요.

    이는 해커가 악성 스크립트를 삽입하여 사용자 몰래 다른 웹사이트의 민감한 데이터를 탈취하는 것을 막기 위한 보안 조치이다.

    ### 1. 네트워크 전송여부: 요청은 서버에 도달한다

    `fetch` 요청은 **일반적으로 네트워크를 통해 대상 서버로 전송**된다.

    - CSRF 방어 실패: `fetch` 요청 자체는 동일한 출처 정책(SOP)의 네트워크 전송 제한을 받지 않으며, 서버에 도달하여 서버 자원을 변경하거나 조회하는 **요청 자체는 성공**한다.
    - 세션 쿠키 전송: 만약 `credentials:’include’` 옵션을 사용했거나 기본적으로 쿠키가 포함되는 요청이라면, 사용자의 세션 쿠키도 함께 전송된다. 이는 CSRF 공격을 막기 위해 서버 측에서 토큰 검증이나 SameSite 쿠키 설정이 필요함을 보여준다.

    ### 2. 응답 사용 가능성: 응답 데이터 접근은 차단된다

    네트워크 요청이 성공적으로 서버에 도달하고 서버가 응답을 보내더라도, 브라우저는 **응답데이터(Headers, Body)**를 스크립트에서 사용하는 것을 엄격하게 차단한다.

    - CORS 실패: 브라우저는 서버로부터 받은 응답 헤더에 현재 요청을 보낸 출처를 허용한다는 내용의 `Access-Control-Allow-Origin` 헤더가 포함되어 있는지 확인한다.
        - 헤더가 없는 경우(일반적): 대상 서버가 CORS 설정을 하지 않았다면, 브라우저는 응답이 악성 스크립트에게 노출되는 것을 막기 위해 해당 응답을 격리시키고 폐기한다.
        - 응답 사용 불가: 악성 스크립트는 `fetch()`가 반환한 **Response 객체에 접근할 수 없으**며, 응답의 상태코드, 헤더, 본문 데이터를 읽을 수 없다.

    ### 3. 브라우저 콘솔 메시지: 명확한 오류가 표시된다

    - 오류 유형: `fetch` 요청이 성공적으로 완료되지 않고 `TypeError: Failed to fetch`와 같은 오류가 발생하거나, 요청은 성공했지만 응답을 읽으려 할 때 CORS 정책 위반에 대한 오류 메시지가 표시된다.
    - 표시내용: "Access to fetch at '대상 URL' from origin '공격 스크립트 출처' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource.”

    이 오류는 JavaScript 코드에는 단순히 **네트워크 요청 실패**로만 전달되어, 악성 스크립트가 민감한 데이터를 탈취하는 것을 **최종적으로 방어**하는 핵심적인 역할을 한다. (데이터는 서버에서 넘어왔지만, 브라우저가 스크립트에게 전달하지 않음)

- SOP가 어떻게 Session Hijacking(세션 하이재킹) 시도를 방지하는지 구체적으로 설명하세요. SOP가 차단하는 것과 허용되는 것(예: 네트워크 요청은 나가지만 응답 데이터에 접근 불가)을 포함하세요.

    SOP는 기본적으로 **’다른 출처에서 온 스크립트는 민감한 데이터에 접근할 수 없다’**는 원칙으로 세션 하이재킹을 방어한다.

    **SOP의 방어 핵심: 응답 데이터 접근 차단**

    세션 하이재킹의 가장 흔한 공격 경로는 **XSS**를 통해 세션 쿠키나 다른 인증정보를 읽어내려고 하는 것이다. SOP는 이 과정의 핵심 단계인 **‘데이터 읽기’**를 차단한다.

    | 구분 | 브라우저 동작 | 공격 방어 메커니즘 |
    | --- | --- | --- |
    | 차단되는 것(방어 성공) | 다른 출처의 스크립트가 민감한 응답 데이터를 읽는 행위 | 해커가 fetch나 XMLHttpRequest로 세션 토큰이 포함된 JSON 응답을 받아도, SOP에 의해 브라우저가 해당 응답 데이터를 스크립트에 넘겨주는 것을 차단한다. 즉, 해커는 토큰을 훔쳐갈 수 없다. |
    | 허용되는 것(방어 실패 아님) | 다른 출처로의 네트워크 요청 전송(특히 단순 요청) | 해커의 스크립트는 요청 자체를 서버로 보낼 수 있다. 이 요청에는 사용자의 세션 쿠키도 자동으로 포함될 수 있다. |

    ### 세가지 측면에서 구체적 설명

    1. **네트워크 요청 전송(허용됨) → CSRF 관련**

    SOP는 **요청을 보내는 것 자체는 막지 않는다.**

    - 동작: 해커의 사이트(B)에 있는 스크립트는 `fetch()`를 사용하여 사용자가 로그인된 은행 사이트(A)로 “나 만 원 이체할래” 라는 요청을 보낼 수 있다.
    - 쿠키 포함: 이 요청에는 브라우저가 자동으로 A 사이트의 **세션 쿠키**를 첨부한다.
    - 결과: 요청은 서버에 도달하며, 이로 인해 서버의 데이터가 변경될 수 있다. 이것이 바로 SOP만으로는 막을 수 없는 CSRF공격이다. 세션 하이재킹 자체가 아니라 **세션 권한을 이용한 요청 위조**이다.
    1. 응답 데이터 접근(차단됨) → 세션 하이재킹 방지

    해커의 궁극적인 목표인 세션 정보를 훔치는 행위는 SOP에 의해 차단된다.

    - 공격시도: 해커는 fetch() 요청을 통해 A 사이트의 사용자 정보를 조회하거나, 세션 토큰을 담고 있는 응답 JSON을 받아오려고 한다
    - SOP의 차단: 브라우저는 응답을 받은 후, 서버의 응답 헤더에 Access-Control-Allow-Origin 헤더(CORS)가 설정되어 있지 않다면, 해커의 스크립트가 응답 본문(response.json() 등)에 접근하는 것을 전면 차단한다.
    - 결론: 해커의 스크립트는 요청이 성공했는지 여부만 알 수 있을 뿐, 세션 쿠키나 민감한 데이터가 포함된 응답 내용을 절대로 읽을 수 없다. 따라서 세션 하이재킹을 위해 필요한 세션 정보 탈취가 불가능해진다.
    1. DOM 접근 및 쿠키 읽기(일부 차단)→XSS 관련
    - 다른 출처의 DOM 접근: SOP는 iframe 등을 통해 **다른 출처의 웹페이지 내용(DOM)**에 스크립트가 접근하는 것을 원천적으로 차단한다.
    - 쿠키 읽기 제한: 쿠키에 `HttpOnly` 속성을 설정하면, 아예 **JavaScript를 통한 쿠키 접근(`document.cookie`)을 막아** 세션 하이재킹을 위한 쿠키 탈취 시도를 무력화할 수 있다. (이는 SOP와 별개의 조치이지만, 세션 보안의 핵십이다.)

    요약: SOP는 악성 스크립트가 세션 정보를 **훔쳐 읽어가는 것**을 막는 강력한 방어선이다. 요청을 보내는 것은 막지 못하지만(CSRF), 그 요청의 결과를 스크립트에 노출시키지 않아 **민감한 정보 유출**을 방지한다.


- 블로그에서 명시한 대로 SOP가 반드시 동일 출처에서만 접근하도록 하는 주요 브라우저 API/리소스 3가지를 쓰고, 각각에 대해 간단한 설명(왜 제한되는지)을 덧붙이세요.
    1. `XMLHttpRequest` 및 `fetch API` 응답 읽기

        **목표: 민감한 정보 탈취 방지**

        다른 출처로 HTTP 요청을 보내는 것 자체는 허용되지만(CSRF의 가능성), 요청에 대한 응답의 본문(Body)이나 상세 헤더를 요청을 보낸 스크립트가 읽는 것은 차단된다. 이는 해커가 사용자 몰래 다른 웹사이트의 로그인 정보나 개인 데이터를 조회하고 훔쳐가는 것을 막기 위함이다.

    2. document.domain을 통한 DOM 접근

        **목표: 웹페이지 조작 방지**

        다른 출처의 **iframe 내부에 있는 문서의 내용(DOM, 즉 HTML 구조)**에 접근하여 내용을 읽거나 수정하는 것을 막는다. 만약 허용된다면, 해커가 자신의 웹사이트에 은행 사이트를 iframe으로 띄워놓고 사용자의 입력 내용을 실시간으로 훔쳐보거나 조작할 수 있게 된다.

    3. 쿠키, 로컬 스토리지 (localStorage), 세션 스토리지(sessionStorage)

        **목표: 사용자 세션 및 영구 데이터 보호**

        스크립트가 다른 출처에 속한 쿠키(특히 세션 쿠키)나 저장소 데이터에 접근하여 읽는 것을 막는다. 이 데이터에는 사용자의 로그인 상태를 유지하는 중요한 토큰이나 개인정보가 포함되어 있어, 다른 출처 스크립트의 접근을 막아 세션 하이재킹이나 데이터 탈취를 방지한다.


    ⇒ SOP는 **데이터 읽기**를 핵심적으로 제한하여, 악성 스크립트가 사용자 몰래 다른 웹사이트의 민감한 정보를 열람하거나 조작하는 것을 원천적으로 막는 보안 장벽 역할을 수행한다.

- SOP와 CSP의 차이를 블로그 내용에 따라 요점 4개(각 항목 1문장)로 정리하세요. (예: 누가 적용하는가, 제어 주체, 설정 가능 여부 등)

    두 정책 모두 XSS 공격 방지를 목표로 하고 있지만, 접근 방식이 다르다.

    |  | CSP(Content Security Policy) | SOP(Same Origin Policy) |
    | --- | --- | --- |
    | 정책 정의 주체 및 성격 | 서버-브라우저 협력 정책 | 브라우저 기본 정책 |
    | 정책 적용 방식 | 개발자가 직접 설정 | 자동 적용 |
    | 적용 범위 및 상세 설정 | 애플리케이션별 맞춤 설정 | 브라우저별 구현 차이 |
    | 관리 및 통제 | 유연한 정책 관리 | 프로그래밍 영역 밖 |

    CSP(Content Security Policy)

    - **서버-브라우저 협력 정책**: 서버에서 정책을 설정하고 브라우저가 실행
    - **개발자가 직접 설정**: HTTP 응답 헤더를 통해 정책 전달
    - **애플리케이션별 맞춤 설정**: 각 서비스의 요구사항에 맞게 조정 가능
    - **유연한 정책 관리**: Report-Only 모드로 단계적 적용 가능

    SOP(Same Origin Policy)

    - **브라우저 기본 정책**: 서버 개입 없이 브라우저가 자체적으로 적용
    - **자동 적용**: 별도 설정 없이 모든 웹페이지에 기본 적용
    - **브라우저별 구현 차이**: 세부 동작이 브라우저마다 약간씩 다름
    - **프로그래밍 영역 밖**: 개발자가 직접 제어할 수 없음

    CSP는 건물주가 정하는 보안규칙이고, SOP는 건물 자체의 구조적 보안 장치라고 생각하면 된다.

- 브라우저에서 SOP 관련 차단 오류를 발견했을 때(예: 콘솔에 “동일 출처 정책으로 인해 ... 차단했습니다” 메시지) 문제 원인 파악을 위한 체크리스트(최소 3항목)를 작성하고, 임시·영구 대응 방안(각 1~2줄)도 제시하세요.
    - [ ]  **출처 3요소 불일치 확인**

        →요청을 보낸 곳과 요청을 받은 곳의 프로토콜, 호스트(도메인), 포트 중 다른 요소가 있는지 확인한다

        이 요소 중 하나라도 다르면 SOP에 의해 차단이 발생한다.(가장 기본적인 원인)

    - [ ]  **요청의 목적 및 타입 확인**

        →차단된 요청이 단순 `GET` 요청인지, 아니면 `POST`나 커스텀 헤더를 포함하는 복잡한 요청인지 확인한다

        복잡한 요청은 브라우저가 본 요청 전에 Preflight(사전 요청)을 보내는데, 여기서 CORS 설정이 없으면 본 요청이 아예 차단될 수 있다.

    - [ ]  **서버 측 CORS 헤더 확인**

        →요청을 받은 서버의 응답 헤더에 `Access-Control-Allow-Origin` 헤더가 포함되어 있는지, 그리고 그 값이 현재 요청 출처를 허용하고 있는지 확인한다.

        SOP의 예외를 허용하는 유일한 방법은 서버가 CORS 헤더를 명시적으로 보내는 것이기 때문.


    - **임시 대응 방안 (개발 및 테스트용)**
        - **브라우저 보안 비활성화**: 개발 단계에서만 사용하도록 Chrome의 보안 기능을 임시로 비활성화하여 브라우저의 CORS 정책 검사를 건너뛴다. (예: `—disable-web-security` 플래그 사용)
    - **영구 대응 방안(서비스 배포용)**
        - **서버 설정 변경**: 요청을 받는 서버(API 서버)의 응답 헤더에 Access-Control-Allow-Origin을 추가하고, 여기에 요청을 보내는 웹사이트의 출처를 명시적으로 허용하도록 설정한다.
        - **프록시 서버 도입**: 클라이언트(브라우저)와 서버 사이에 프록시 서버를 두어, SOP가 적용되는 ‘다른 출처’ 문제를 **‘같은 출처’ 문제**로 우회하여 해결한다.


# **📚 블로그 읽고 교차 출처 리소스 공유(CORS) 정리해보기 🍠**

CORS란 브라우저에서 **다른 출처**에 있는 리소스를 사용할 수 있는 규칙이다.

핵심 용어 정리

1. 출처(Origin)

    URL의 프로토콜, 호스트, 포트 세 가지를 조합한 것

    - 이 세가지가 모두 같으면 출처가 같다고 표현한다
    - 하나라도 다르면 출처가 다르다고 표현한다
2. 교차 출처 요청(Cross-Origin Request)

    다른 출처의 자원을 사용하기 위해 네트워크 요청을 만드는 것

    마치 다른 동네에 있는 편의점에 가서 물건을 사려는 것과 같다. 내가 사는 동네가 아니기 때문에, 그 편의점 주인이 “당신에게 팔아도 되나요?”라고 먼저 확인하는 것

    브라우저는 실행중인 애플리케이션의 출처와 다른 출처의 자원을 요청하면 이를 차단한다. 앞서 공부한 `동일 출처 정책`이 동작하기 때문이다.

    이를 허용하려면 서버와 특별한 약속을 지켜야 하는데, 그것이 바로 `CORS`다.


- 브라우저에서 `http://localhost:8080` 애플리케이션이 `http://localhost:8081/resource.json`을 요청했을 때, 네트워크 요청과 응답은 어떻게 처리되며, 브라우저가 응답 본문을 사용하지 못하는 이유는 무엇인가요?

    ![image.png](attachment:ca0875db-2816-4d59-9da7-e7d3b1e7f750:image.png)

    `8081` 포트로 요청을 보내고, 응답도 받았지만, 스크립트에는 응답 본문을 사용할 수 없음이라고 표기된다.

    원인: Access-Control-Allow-Origin CORS 헤더가 없기 때문이다.

    브라우저는 다른 출처로 요청을 보낼 때:

    1. 응답 헤더 중 Access-Control-Allow-Origin을 확인한다
    2. 이 헤더에 현재 출처가 포함되어 있는지 검사한다
    3. 서버가 해당 출처를 허용했는지 판단한다

    서버 `8081`은 아직 이에 대해 명시하지 않았기 때문에, 브라우저는 자원 사용을 허용하지 않은 것으로 판단하여 네트워크 오류를 발생시킨 것이다.

- 게 설정해야 하나요? 글의 예시 코드를 참고해 헤더 이름과 값까지 구체적으로 쓰세요.

    서버 8081이 다른 출처에게 자원을 공유하려면 `Access-Control-Allow-Origin` 헤더를 설정해야한다.

    이번에는 정상적으로 네트워크 요청이 전송되었다. 서버 8081로 요청을 보냈고, 응답도 받았으며, 브라우저가 이 응답을 차단하지도 않았다.

    ```jsx
    const handler=(req,res)=>{
      //localhost:8080 출처를 허용
      res.setHeader('Access-Control-Allow-Origin','http://localhost:8080');
      static(path.join(__dirname,'public'))(req,res);
      };
    ```

    ⇒응답 헤더의 `Access-Control-Allow-Origin`에 현재 애플리케이션이 실행되는 `localhost:8080`이 명시되어 있기 때문이다.

    브라우저는 “서버가 `8080`번 포트의 출처도 이 자원을 사용할 수 있다고 허락했구나”라고 판단하여 응답 사용을 허용한 것이다.

- 단순 요청으로 분류되기 위해서는 어떤 두 가지 조건을 만족해야 하나요? 또한 `GET /resource.json` 요청이 단순 요청에 해당하는 이유를 설명하세요.

    교차 출처 요청 중에도 특정 조건을 만족하는 요청을 **단순 요청(Simple Request)**라고 한다.

    단순 요청의 조건

    1. **허용된 HTTP 메소드 사용**
        - `GET`
        - `POST`
        - `HEAD`
    2. **안전한 헤더 사용**

        **안전한 헤더**: 교차 출처 요청에서 사용할 수 있는 헤더

        브라우저는 다른 출처의 자원을 사용하기 위해 HTTP 요청을 만들 때 `Origin` 헤더에 현재 출처를 실어서 보낸다.

        요청 헤더의 `Origin`에 현재 애플리케이션이 실행된 URL이 담겨있는 것: 현재 출처

        브라우저가 서버에게 “나의 출처는 이곳이니, 당신의 자원을 사용해도 되나요?”라고 묻는 것.

        서버는 자원을 제공하겠다는 의미로 응답 헤더에 Access-Control-Allow-Origin을 포함시킨다

        응답을 받은 브라우저는 이 헤더에 **자신의 출처가 포함되어 있는지** 확인한다. 포함되어 있다면 “서버가 자원 사용을 허락했다”고 판단하여 응답을 사용한다.

        안전한 헤더 목록:

        - `Accept`
        - `Accept-Language`
        - `Content-Language`
        - `Content-Type`
        - `Range`

        브라우저에게 교차 출처 요청을 할 때 이외의 헤더를 사용하면, 출처 확인과 마찬가지로 헤더 사용 여부도 서버에게 확인받아야 한다.

- 브라우저에서 `X-Goguma`라는 커스텀 헤더를 추가했을 때 왜 차단이 발생하나요? 이 문제를 해결하기 위해 서버에서 추가해야 하는 응답 헤더와 값은 무엇인가요?

    ```tsx
    const init = async () => {
      const response = await fetch('http://localhost:8081/resource.json', {
        // 안전하지 않은 헤더를 보냄
        headers: {
          'X-Goguma': 'goguma',
        },
      });
      const data = await response.text();
      const jsonElement = document.createElement('pre');
      jsonElement.textContent = data;
      document.body.appendChild(jsonElement);
    };
    document.addEventListener('DOMContentLoaded', init);
    ```

    브라우저를 새로 고침하면
    커스텀 헤더를 보냈지만, 동일 출처 정책으로 인해 응답을 받을 수 없음

`동일 출처 정책`으로 인해 원격 리소스가 차단되었다. `Access-Control-Allow-Headers`에 헤더 `X-Goguma`가 허용되지 않았다고 표시된다.

안전하지 않은 헤더를 사용하려면 서버가 `Access-Control-Allow-Headers` 응답 헤더에 허용할 헤더 이름을 명시해야 한다.

```tsx
const handler=(req,res)=>{
  res.setHeader('Access-Control-Allow-Origin','http://localhost:8080');
  //X-Goguma 헤더를 허용
  **res.setHeader('Access-Control-Allow-Headers','X-Goguma');**
  static(path.join(__dirname,'public'))(req,res);
  };
````

서버를 재시작하면 이제 정상적으로 요청이 처리된다

- `PUT` 요청을 보낼 때 브라우저는 왜 먼저 `OPTIONS` 요청을 보내나요? 이때 브라우저가 보내는 헤더와 서버가 응답해야 하는 헤더를 각각 쓰고, 사전 요청과 실제 요청이 어떻게 이어지는지 간단히 서술하세요.
  `PUT` 요청은 데이터를 서버에 새로 만들거나 덮어쓰는 요청이다. 이는 서버의 자원에 ‘위험한(side effect)’ 변경을 일으킬 수 있는 행위로 간주된다.
  웹 보안의 핵심인 동일 출처 정책은 다른 출처로 요청을 보내는 것을 허용하지만, 악성 스크립트가 사용자의 권한을 도용하여 서버 데이터를 함부로 변경하는 것을 막아야 한다.
  따라서, 브라우저는 `PUT`,`DELETE` 또는 커스텀 헤더를 포함하는 `POST`와 같은 복잡한 요청(Complex Request)을 보낼 때, 다음과 같은 단계를 거친다
  1. **사전 확인**:”이 서버는 내가 다른 출처에서 보낸 이런 종류의 위험한 요청을 받아줄 준비가 되어있나요?”라고 먼저 물어본다. 이 질문이 바로 `OPTIONS` 요청(사전요청)이다.
  2. **안전 확인**: 서버가 명시적으로 “응, 허용할게”라고 응답해야만, 브라우저는 안심하고 본 요청(`PUT` 요청)을 보낸다.

  3. **브라우저가 OPTIONS(사전 요청) 시 보내는 헤더**

     | 헤더 이름                      | 설명                                                             |
     | ------------------------------ | ---------------------------------------------------------------- |
     | Origin                         | 요청을 보낸 웹 페이지의 출처를 알린다 (예: `https://client.com`) |
     | Access-Control-Request-Method  | 실제 보낼 요청의 HTTP 메서드를 알린다. (예: `PUT`)               |
     | Access-Control-Request-Headers | 실제 요청에 포함될 사용자 정의 헤더 목록을 알린다                |

  4. **서버가 OPTIONS(사전 요청)에 응답해야 하는 헤더**

     서버는 브라우저의 질문에 응답하여, “허용할게”라는 정보를 포함시켜야 한다. 상태 코드는 `200OK`를 사용한다.

     | 헤더 이름                    | 설명                                                                                                        |
     | ---------------------------- | ----------------------------------------------------------------------------------------------------------- |
     | Access-Control-Allow-Origin  | 요청을 허용할 출처를 알린다. (예: `https://client.com` 또는 `*`(모두 허용))                                 |
     | Access-Control-Allow-Methods | 서버가 허용하는 HTTP 메서드를 알린다. (반드시 `PUT`을 포함해야 함)                                          |
     | Access-Control-Allow-Headers | 서버가 허용하는 사용자 정의 헤더를 알린다                                                                   |
     | Access-Control-Max-Age       | 이 사전 요청 응답을 캐시할 시간(초)을 브라우저에 알려, 같은 요청에 대해 재차 `OPTIONS`를 보내지 않도록 한다 |

     1. 사전 요청과 실제 요청의 흐름

        | 단계  | 요청 유형            | 주체     | 동작                                                                                                                                 |
        | ----- | -------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------ |
        | 1단계 | 사전 요청(Preflight) | 브라우저 | 서버에게 `OPTIONS` 요청을 보내며, `Access-Control-Request-Method:PUT`과 `Origin` 정보를 담아 허용 여부를 문의한다                    |
        | 2단계 | 사전 요청 응답       | 서버     | `Access-Control-Allow-Methods:PUT` 등의 허용 정보를 담아 응답한다. 만약 허용하지 않으면 CORS 오류가 발생하며 3단계로 넘어가지 않는다 |
        | 3단계 | 실제 요청            | 브라우저 | 서버의 응답을 확인하고 안전하다고 판단되면, `PUT` 메서드를 사용해 본격적인 데이터 요청을 보낸다                                      |
        | 4단계 | 최종 응답            | 서버     | `PUT` 요청에 따라 데이터를 처리하고 결과를 브라우저에게 응답한다.                                                                    |

- **RBAC**의 한계에 대해 설명해주세요.

  1. **역할 폭발(Role Explosion) 문제**

     조직이 커지거나 보안 요구사항이 복잡해질수록 관리해야 할 역할의 수가 지나치게 증가하는 현상

     - 원인: 특정 사용자나 부서에 아주 미세한 차이의 권한이 필요할 때, 기존 역할로는 처리할 수 없어 새로운 역할을 계속 만들어내기 때문
     - 예시: ‘영업팀’ 역할이 있지만, 특정 사용자에게는 ‘VIP 고객 데이터’만 볼 수 있도록 허용해야 할 때, ‘영업팀-VIP용’이라는 새로운 역할이 필요해진다.
     - 결과: 수백, 수천 개의 역할이 생겨나 관리가 복잡해지고, 결국 어떤 역할이 어떤 권한을 가지는지 파악하기 어려워진다

  2. **컨텍스트(Context) 및 속성 기반 제어 불가**

     RBAC는 ‘누가 무엇을 할 수 있는가?’(역할)만 정의할 수 있고, ‘어떤 상황에서(언제, 어디서, 어떤 데이터에)’ 접근을 허용할지는 정의할 수 없다.

     - 시간제약: “재무팀 직원이라도 업무시간(오전 9시~6시)이 아니면 급여 정보에 접근할 수 없다”
     - 위치 제약: “관리자라도 회사 외부 IP에서는 서버 설정에 접근할 수 없다”
     - 데이터 속성 제약: “의사는 자신이 담당하는 환자의 기록만 볼 수 있다”

     결과: RBAC는 이러한 상황적 요인을 고려하지 못하고, 역할만 있으면 무조건 접근을 허용하기 때문에 유연하고 정교한 보안 정책을 구현할 수 없다.

  3. **최소 권한 원칙 위반 가능성**

     RBAC는 역할 단위로 권한을 부여하기 때문에, 사용자에게 실제 필요한 것보다 더 많은 권한이 보여될 가능성이 높다

     - 원인: 역할을 할당할 때, 관리 편의상 가장 포괄적인 역할을 부여하는 경우가 많다
     - 예시: 직원이 실제로는 데이터 조회 권한만 필요한데, ‘관리자’ 역할에 포함되어 있다는 이유만으로 데이터 수정 및 삭제 권한까지 함께 부여받게 된다
     - 결과: 필요 이상의 권한이 부여되면 해킹 공격에 취약해지거나, 직원의 실수로 인해 시스템에 심각한 문제가 발생할 위험이 커짐

  4. **정적인(Static) 관리의 어려움**
     - 원인: 사용자들은 수시로 부서를 옮기고, 직무가 바뀌고, 프로젝트 권한이 필요해진다.
     - 결과: 조직의 변화 속도를 RBAC 시스템이 따라가지 못해, 접근 권한 부여나 회수가 늦어지는 **‘권한 격차(Permission Gap)’가 발생한다.**(퇴사했는데도 권한이 남아있는 등)

- **ABAC**으로의 전환, 어떤 '기준'이 적절할까요?
  **“우리 조직의 보안 정책이 너무 복잡하고 동적이어서, 단순히 ‘역할’만으로는 ‘누가’, ‘어떤 상황’에서 ‘어떤 데이터’에 접근해야 하는지를 충분히 정의할 수 없다.”**라는 질문에 **“네”**라고 답할 수 있을 때 가장 적절하다.
  1. 정교한 ‘컨텍스트 기반’ 통제가 필요한가?

     가장 중요한 기준은 **접근 제어를 ‘누가’에서 ‘언제, 어디서, 왜, 어떻게’로 확장**해야 할 필요가 있을 때이다.

     | 구분 | RBAC의 한계                                                                           |
     | ---- | ------------------------------------------------------------------------------------- |
     | 기준 | 접근 시점/환경을 제어해야 할 때                                                       |
     | 예시 | “프로젝트 관리자 역할이라도 주말(시간 속성)에는 기밀 문서에 접근할 수 없다.”          |
     | 판단 | 단순한 역할 할당만으로는 보안 요구사항을 충족할 수 없을 때 ABAC 전환을 고려해야 한다. |

  2. ‘역할 폭발’ 현상으로 관리가 불가능한가?

     관리해야할 역할의 수가 너무 많아져(수백, 수천 개) 권한 관리가 비효율적이거나 오류가 잦아질 때 ABAC가 대안이 된다

     | 구분 | RBAC의 한계                                                                                                                                                                                                 |
     | ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
     | 기준 | 역할 수가 기하급수적으로 증가하여 관리가 어려울 때                                                                                                                                                          |
     | 예시 | 고객 수가 100만 명일 때, RBAC는 고객 ID별로 조회 권한을 주기 어려워 포괄적 권한을 줄 수밖에 없다. ABAC는 “사용자 ID와 문서의 작성자 ID가 같을 때만 접근 허용(속성 일치)”이라는 단 하나의 정책으로 처리한다. |
     | 판단 | 미세한 권한 차이를 위해 새로운 역할을 계속 만들고 있다면, 속성 기반의 단일 정책으로 통합하기 위해 ABAC로 전환해야한다                                                                                       |

  3. 리소스(데이터) 접근이 ‘개인화’되어야 하는가?

     접근할 리소스(데이터) 자체가 사용자, 소유자, 상태 등의 속성에 따라 달라져야 할 때 ABAC가 필요하다

     | 구분 | RBAC의 한계                                                                                                                                   |
     | ---- | --------------------------------------------------------------------------------------------------------------------------------------------- |
     | 기준 | 데이터의 속성에 따라 접근 권한이 달라져야 할 때                                                                                               |
     | 예시 | “의사는 담당 의사(사용자 속성)와 환자 기록의 담당 의사 필드(리소스 속성)가 일치하는 기록에만 접근할 수 있다”                                  |
     | 판단 | 데이터의 소유권, 민감도, 현재 상태 등 데이터 자체의 특성을 기반으로 접근을 제어하여 최소 권한 원칙을 엄격하게 적용하려면 ABAC로 전환해야 한다 |
- 어떤 서비스 영역에 **RBAC**을 남겨두고, **ABAC**을 도입하시겠어요?
  **정적이고 광범위한 접근**이 필요한 영역에는 **RBAC**를 남겨두고, **동적이고 미세한 제어**가 필요한 영역에 **ABAC**를 도입한다.
  1. **RBAC를 남겨둘 서비스 영역: 기본 관리 및 인프라 접근**

     **“누가 무엇을 할 수 있는가”**를 명확하게 정의할 때 가장 효율적. 권한이 **자주 변하지 않고**, 역할에 따라 **명확하게 구분**되는 영역에 적합

     | 서비스 영역                 | 적용 이유(RBAC의 장점)                                                                                  | 예시 권한                                                         |
     | --------------------------- | ------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------- |
     | 시스템 관리 및 인프라 접근  | 역할이 정적이고, ‘서버 관리자’처럼 광범위한 권한이 필요한다. 역할이 명확하기 때문에 관리 복잡도가 낮다. | 서버 접속(SSH), 데이터베이스 접근, 인프라 배포/수정 권한          |
     | 애플리케이션 기본 기능 접근 | 모든 사용자에게 일률적으로 적용되는 기본적인 ‘읽기/쓰기’ 권한이 필요하다                                | 일반 사용자 로그인, 게시판 조회, 일반적인 설정 변경               |
     | 운영 및 개발팀 권한         | 팀이나 직무에 따라 권한이 명확하게 구분되어야 한다                                                      | '개발자' 역할 (코드 저장소 접근), '운영자' 역할 (로그 조회, 배포) |

  2. ABAC을 도입해야 할 서비스 영역: 민감 데이터 및 개인화 접근

     ‘어떤 상황에서’, ‘어떤 속성을 가진 데이터에’ 접근할 수 있는지를 정교하게 제어할 때 필요.

     권한이 동적이고, 데이터의 속성(소유자, 상태 등)에 따라 접근이 달라져야 하는 영역에 적합

     | 서비스 영역                           | 적용 이유(ABAC의 필요성)                                                                              | 예시 권한                                                                                                                                    |
     | ------------------------------------- | ----------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
     | 개인화된 민감 데이터 접근             | **데이터의 소유자 속성**을 확인하여 접근을 제어해야 한다. RBAC의 '역할 폭발' 문제를 해결한다.         | 은행/금융: “사용자가 자신의 계좌 정보(리소스 속성)에만 접근 허용” 헬스 케어: “의사가 자신이 담당하는 환자의 기록(리소스 속성)에만 접근 허용” |
     | 동적/상황 기반 거래 및 승인           | **동적/상황 기반 거래 및 승인\***접근 환경(시간, 위치, 위험도)\*\*에 따라 실시간으로 제어가 필요하다. | 결제 승인자 역할이라도 해외 IP(환경 속성)에서 업무 시간 외(시간 속성)에는 특정 금액 이상의 거래(리소스 속성)를 승인할 수 없다                |
     | 대규모 멀티 테넌트(Multi-Tenant) 환경 | 하나의 시스템을 여러 고객(테넌트)이 공유할 때, 테넌트 ID 속성으로 데이터 접근을 격리한다.             | 사용자에게 자신이 속한 테넌트의 데이터(리소스 속성)만 보이도록 제한                                                                          |
- 여러분들은 다른 부서에서 요청을 받았을 때 어떤식으로 행동하실껀가요?
  다른 부서에서 요청을 받으면, 먼저 **요청의 내용과 목적**을 명확히 이해하고 **처리 가능 여부**를 판단한다.
  가능하다면 관련 정보를 제공하거나 작업을 수행하며, 제 **권한 밖의 요청**이라면 해당 부서에 **이유와 함께 적절한 담당자를 안내**할 것 같다.
