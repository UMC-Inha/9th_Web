Referential Equality (참조 동일성)

참조 동일성이란 두 변수가 **같은 객체(같은 메모리 주소)**를 가리키고 있는지를 의미한다.
값이 같은지를 보는 것이 아니라, 완전히 같은 인스턴스를 공유하는지를 판단하는 개념이다.

자바스크립트에서

primitive 타입은 값 자체를 비교하고

object, array, function 같은 참조 타입은 **참조(주소)**를 비교한다.

따라서 내용이 같아 보이는 객체라도 새로 생성되면 서로 다른 참조가 되며, === 비교 결과는 false가 된다.

===와 Object.is는 객체 비교 시 모두 참조 동일성을 기준으로 동작한다.

참조 동일성이 중요한 이유

객체를 여러 변수가 공유하면 한 곳에서의 변경이 다른 곳에 사이드 이펙트를 일으킬 수 있다.
이를 방지하기 위해 React를 포함한 많은 프레임워크에서는 불변성(immutable) 패턴을 사용한다.

불변 패턴에서는

기존 객체를 수정하지 않고

변경 시 항상 새 객체를 생성한다.

이 방식의 핵심 장점은

참조가 바뀌었는지 여부만으로도 변경 여부를 판단할 수 있다는 점이다.

React는 이 특성을 이용해 리렌더링 여부, effect 재실행 여부를 판단한다.

React 렌더링 최적화와 Referential Equality의 관계

React는 값의 “내용”이 아니라 참조가 변경되었는지를 기준으로 렌더링 여부를 판단한다.

객체, 배열, 함수는 렌더링 때마다 새로 생성되면

실제 로직 변화가 없어도

참조가 달라졌다고 인식되어 리렌더링이 발생한다.

따라서 렌더링 최적화의 핵심은
불필요하게 참조를 바꾸지 않고, 실제 변경이 있을 때만 참조를 바꾸는 것이다.

useCallback

useCallback은 함수(콜백)를 메모이제이션하는 Hook이다.
렌더링이 반복되더라도 의존성 배열이 변하지 않으면 같은 함수 참조를 재사용한다.

핵심 개념

함수도 객체이기 때문에, 매 렌더링마다 새로 생성되면 참조가 바뀐다.

useCallback은 이 불필요한 참조 변경을 막아준다.

동작 원리

의존성 배열이 변경되면 → 새로운 함수 생성

의존성 배열이 같으면 → 기존 함수 참조 재사용

사용하는 이유

props로 전달되는 함수의 참조를 안정적으로 유지

memo로 감싼 자식 컴포넌트의 불필요한 리렌더링 방지

useEffect에서 함수 의존성으로 인한 반복 실행 방지

주의점

내부에서 사용하는 state, props는 반드시 의존성 배열에 포함해야 한다.

그렇지 않으면 stale closure(낡은 값 캡처) 문제가 발생할 수 있다.

단순한 컴포넌트에서 남용하면 오히려 코드 복잡도와 비교 비용만 증가한다.

stale closure 문제

콜백 함수는 생성 시점의 state와 props를 클로저로 기억한다.
의존성이 바뀌지 않아 콜백이 재생성되지 않으면, 이후 변경된 값을 반영하지 못하는 문제가 발생한다.

해결 방법은

의존성 배열을 정확히 작성하거나

함수형 state 업데이트를 사용하는 것이다.

memo

memo는 함수형 컴포넌트를 메모이제이션하는 기능이다.
부모가 리렌더링되더라도, 전달된 props가 이전과 같으면 자식 컴포넌트의 렌더링을 건너뛴다.

특징

props 비교는 **얕은 비교(shallow compare)**를 사용한다.

객체나 함수는 참조가 같을 때만 동일하다고 판단한다.

사용하는 이유

부모의 리렌더링이 자식에게 불필요하게 전파되는 것을 방지

렌더링 비용이 큰 컴포넌트에서 성능 개선 효과

언제 쓰면 좋은가

부모는 자주 리렌더링되지만 자식 props는 거의 변하지 않을 때

리스트 아이템처럼 반복 렌더링되는 컴포넌트

렌더링 비용이 큰 컴포넌트

언제 쓰지 말아야 하는가

렌더링 비용이 거의 없는 단순 컴포넌트

props가 매번 자주 변경되는 구조

객체/함수를 props로 넘기면서 참조를 안정적으로 관리하지 않는 경우
