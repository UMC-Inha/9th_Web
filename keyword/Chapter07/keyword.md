# useMutation의 핵심 옵션

`useMutation`은 단순히 데이터 변경 요청만 수행하는 훅이 아니라,  
요청 전 → 요청 중 → 성공/실패 이후의 전체 과정을 세밀하게 제어할 수 있는 다양한 기능을 제공합니다.

## 필수 옵션 — 변경 작업 정의

### `mutationFn`

- 서버에 데이터를 보내는 비동기 함수를 지정합니다.
- 예: POST, PUT, DELETE 요청을 담당
- 어떤 데이터를 어떻게 변경할지 명확히 정의해야 하므로 핵심 옵션입니다.

## 라이프사이클 옵션 — 요청 흐름 제어

요청이 실행되는 동안 특정 시점에 자동으로 실행되는 콜백 함수들입니다.

### `onMutate`

- 요청이 서버로 보내지기 **직전** 호출됩니다.
- 주 사용처: 낙관적 업데이트(Optimistic Update)
  - UI를 서버 응답보다 먼저 변경
  - 에러 시 롤백할 데이터를 저장

### `onSuccess`

- 요청이 성공적으로 완료되었을 때 호출됩니다.
- 주 사용처:
  - 서버 상태와 UI를 동기화
  - `invalidateQueries`로 필요한 데이터만 새로고침

### `onError`

- 요청 중 에러가 발생했을 때 호출됩니다.
- 주 사용처:
  - 사용자에게 오류 알림
  - `onMutate` 단계에서 미리 수정한 UI를 원래 상태로 롤백

### `onSettled`

- 성공이든 실패든, 요청이 완전히 끝났을 때 호출됩니다.
- 주 사용처:
  - 로딩 상태 해제
  - 정리(clean-up) 작업

## 설정 옵션 — 제어 및 안정성 강화

### `retry`

- 요청 실패 시 자동 재시도 횟수를 지정합니다.
- 네트워크 오류 등 일시적 장애 시 유용

### `retryDelay`

- 재시도 시도 간의 대기 시간을 설정합니다.
- 서버에 무리를 주지 않도록 간격을 조정

### `throwOnError`

- 에러를 내부에서 처리하지 않고 외부로 던질지 여부를 설정합니다.
- `ErrorBoundary` 등 상위 단에서 일괄 처리할 때 사용

### `meta`

- mutation에 관련한 추가 정보를 담는 객체
- 주 사용처:
  - 로깅
  - 디버깅
  - 커스텀 처리

# 낙관적 업데이트(Optimistic Update)

## 낙관적 업데이트가 효율적인 상황

낙관적 업데이트는 요청 성공 가능성이 높고,  
사용자 경험(UX)을 빠르게 개선하는 것이 중요한 상황에서 특히 효과적입니다.

서버 응답을 기다리지 않고 UI를 먼저 갱신함으로써  
사용자가 즉각적인 반응을 느끼도록 하는 것이 핵심입니다.

### 특징

- 요청이 대부분 성공할 것으로 예상됨
- UI 반응 속도가 중요한 경우
- 서버 응답 지연이 있더라도 UX를 우선하고 싶은 경우
- 실패 시 롤백이 비교적 쉬운 경우

### 예시

- 쇼핑몰 장바구니에 상품 추가
- 댓글 작성 후 즉시 리스트에 표시
- 좋아요 / 북마크 UI 반영
- 투표 기능

정리: "빠른 UX가 중요하고, 실패해도 되돌리기 쉬운 경우 → 낙관적 업데이트 사용"

## 낙관적 업데이트를 피해야 하는 상황

낙관적 업데이트는 편리하지만,  
데이터 정확성이 UX보다 중요한 상황에서는 위험할 수 있습니다.

요청 실패 시 데이터 일관성과 신뢰성이 무너질 수 있으며,  
롤백이 어렵다면 UI/서버 데이터가 불일치하는 문제도 발생할 수 있습니다.

### 특징

- 실패 가능성이 높은 요청
- 성공/실패 여부가 비즈니스적으로 중요한 경우
- 롤백이 어렵거나 불가능한 경우
- 서버 데이터 정확성이 매우 중요한 경우

### 예시

- 은행 송금, 잔액 표시
- 주식 매수/매도
- 결제 처리
- 주요 비즈니스 데이터 변경 (예: 주문 확정, 취소 등)

정리: "정확성이 최우선이고 롤백이 어렵다면 → 낙관적 업데이트 피함"

# 낙관적 업데이트(Optimistic Update) 패턴과 롤백 전략

## 낙관적 업데이트를 도입하는 이유

낙관적 업데이트는 사용자가 경험하는 **대기 시간(Latency)**과 답답함을 해결하기 위해 도입하는 UI/UX 패턴입니다.  
서버의 응답 속도가 UI 반응 속도를 저해하는 문제를 해결하는 것이 핵심 목적입니다.

### 기존 방식의 문제점

1. **사용자 액션:** 사용자가 '좋아요' 버튼 클릭
2. **서버 요청:** 애플리케이션이 API 호출
3. **대기:** 사용자는 서버 응답을 기다리며 로딩 스피너를 확인
4. **UI 업데이트:** 서버 응답을 받아야 '좋아요' 아이콘 변경

- 문제: 네트워크 속도가 느리면 1~2초 이상 기다려야 하며, 사용자에게 "내가 누른 게 반영됐나?"라는 불확실성과 답답함을 줌
- 결과: 서비스의 응답성과 UX 저하

### 낙관적 업데이트의 해결책

1. **사용자 액션:** '좋아요' 버튼 클릭
2. **즉각 UI 업데이트:** 서버 응답을 기다리지 않고 UI 반영
3. **서버 요청 (백그라운드):** 실제 API 호출

- 장점: 사용자는 즉각적인 시각적 피드백을 받음
- UX가 빠르고 자연스러워짐, 네트워크 속도와 무관하게 앱이 반응

## 2. TanStack Query 기반 흐름

`onMutate → mutate → onError → onSettled` 순서로 진행됩니다.

### onMutate: UI 즉시 업데이트 및 롤백 준비

1. **쿼리 취소 (`queryClient.cancelQueries`)**
   - 진행 중인 동일 데이터 refetch 취소
   - 데이터 충돌(Race Condition) 방지
2. **이전 데이터 저장 (`queryClient.getQueryData`)**
   - 실패 시 롤백용으로 기존 데이터 저장
3. **캐시 수동 업데이트 (`queryClient.setQueryData`)**
   - 서버 응답을 기다리지 않고 UI 즉시 변경
4. **컨텍스트 반환**
   - 롤백용 `previousTodos` 등 데이터를 반환, `onError`와 `onSettled`에서 사용

### mutate: 실제 서버 요청

- `mutationFn` 실행
- 백그라운드에서 서버 통신

### onError: 요청 실패 시 롤백

1. `context`에서 이전 데이터 접근
2. `queryClient.setQueryData`로 캐시 원상복구
3. UI 즉시 롤백
4. 사용자에게 실패 알림 표시 가능

### onSettled: 최종 동기화

1. 성공/실패 관계없이 호출
2. `queryClient.invalidateQueries`로 서버 데이터와 클라이언트 캐시 동기화
3. 낙관적 업데이트로 반영된 데이터 보완 또는 검증

## 3. ToDo 생성 및 좋아요 토글 롤백 전략

### 3.1 ToDo 생성(Create)

#### 낙관적 업데이트 (`onMutate`)

1. 입력된 `newTodo`에 임시 ID 부여
2. `queryClient.getQueryData`로 현재 ToDo 목록(`previousTodos`) 저장
3. `queryClient.setQueryData`로 임시 ToDo UI에 추가
4. 컨텍스트에 이전 데이터 저장

#### 실패(Failure) 롤백 (`onError`)

- 서버 500 오류, 네트워크 끊김
- `context.previousTodos`로 캐시 원복
- UI에서 임시 항목 사라짐
- 피드백: "ToDo 생성에 실패했습니다."

#### 충돌(Conflict) 롤백 (`onError`)

- 409 Conflict 오류
- `context.previousTodos`로 즉시 롤백
- 피드백: "이미 동일한 요청이 처리되었습니다."

### 3.2 좋아요 토글(Like Toggle)

#### 낙관적 업데이트 (`onMutate`)

1. `queryKey`로 특정 게시글 캐시(`previousPost`) 가져오기
2. `isLiked` 값 토글, `likeCount` +1/-1
3. UI 즉시 변경

#### 실패(Failure) 롤백 (`onError`)

- 서버 500 오류, 네트워크 문제
- `context.previousPost`로 캐시 복원
- UI 원래 상태로 롤백
- 피드백: "좋아요 처리에 실패했습니다."

#### 충돌(Conflict) 롤백 (`onError`)

- 404 오류: 다른 기기에서 이미 좋아요 해제
- 전략: `context` 롤백 대신 `queryClient.invalidateQueries` 호출
- 서버에서 최신 데이터 refetch
- UI 서버 데이터와 동기화
