# 어떤 상황에서 **낙관적 업데이트(OptimisticUpdate)가 효율적일까요? 🍠**

---

> **낙관적 업데이트(OptimisticUpdate)가 효율적인 상황은 언제일까요? 🍠**

<aside>
📌

### 1. **사용자에게 빠른 피드백이 가장 중요할 때** (속도 개선)

서버와 통신하는 데 걸리는 지연 시간(Latency)을 사용자에게 느끼게 하고 싶지 않을 때 유용하다.

- **예시:** 소셜 미디어의 '좋아요'나 **'팔로우/구독'** 버튼, 간단한 **체크리스트의 항목 완료** 표시
  - 이러한 기능은 사용자에게 **즉각적인** 만족감을 주는 것이 중요하며, 서버 오류가 발생할 확률은 매우 낮다. 사용자가 즉시 결과물을 확인하고 다음 행동으로 넘어갈 수 있게 해준다.

### 2. **성공 확률이 매우 높을 때** (오류 발생률이 낮을 때)

서버에서 요청을 처리하는 과정이 복잡하지 않아 **실패할 가능성이 거의 없는** 단순한 동작일 때 효율적이다.

- **예시:** **간단한 설정 변경** (예: 다크 모드/라이트 모드 전환), **메시지 보내기** (네트워크 연결이 안정적이라는 전제 하에).
  - 서버의 응답을 기다리느라 시간을 낭비할 필요가 없다. 대부분 성공하므로 사용자는 빠른 속도의 이득을 누릴 수 있다.

### 3. **잠시 동안의 불일치가 허용될 때** (데이터 무결성의 중요도가 상대적으로 낮을 때)

업데이트가 실패했을 경우, 잠시 잘못된 정보가 표시되더라도 **치명적인 문제가 발생하지 않는** 경우에 적합하다.

- **예시:** **댓글 달기** 또는 **게시글 작성**. - 사용자가 댓글을 작성하면 화면에 바로 보여주고, 서버 통신에 실패하면 그때 사용자에게 "댓글 작성에 실패했습니다"라고 알려주고 화면에서 삭제하면 된다. 이 경우 잠깐의 불일치가 큰 문제가 되지 않는다.
</aside>

> **낙관적 업데이트(OptimisticUpdate)를 피해야 하는 상황 언제일까요? 🍠**

<aside>
📌

- **데이터 무결성이 매우 중요할 때:** 돈이 오가는 금융 거래, 재고 관리가 필요한 쇼핑몰 결제 등 **단 한 순간의 불일치**도 큰 손해를 유발할 수 있는 상황.
- **실패 확률이 높을 때:** 서버 부하가 높거나, 복잡한 유효성 검사(Validation)가 필요한 상황. (예: 파일 업로드, 복잡한 폼 제출)
</aside>

- **낙관적 업데이트(OptimisticUpdate)**를 왜 도입해야 하는지, 이 패턴이 해결하려는 문제를 실제 서비스 맥락에서 설명해보세요.
  1. 문제: 지연시간으로 인한 사용자 경험 저하

     인터넷 서비스에서 사용자의 액션이 서버까지 왕복하는 데 걸리는 시간을 지연시간이라고 하는데, 모바일 환경이나 네트워크가 불안정한 환경에서 더 길어진다.

     지연시간이 UX에 미치는 부정적인 영향

     - 피드백 지연: 사용자가 버튼을 눌렀는데, 서버 응답을 기다리느라 아무런 변화가 없거나 로딩스피너가 빙글빙글 도는 것을 봄→ 서비스가 느리다고 느끼게 만듦
     - 불필요한 인지 부하: 사용자에게 “내가 클릭한 것이 정상적으로 작동하고 있는가?”라는 의문을 갖게하고, 불필요하게 서버의 응답을 기다리게 함
     - ‘즉각성’ 결여: 현대의 사용자들은 모든 서비스에서 즉각적인 피드백을 기대함. 작은 액션(좋아요, 팔로우 등)에 대해 1초라도 기다려야 한다면 서비스의 품질이 낮다고 느끼게 됨

  2. 해결책: 즉각적인 시각적 피드백 제공

     낙관적 업데이트는 지연시간을 눈속임으로 해소함

     - 사용자는 자신이 원하는 결과를 지연시간 없이 즉시 화면에서 확인
     - 이로 인해 서비스가 매우 빠르고 반응성이 좋다고 느낌
     - 실제 서버 통신은 백그라운드에서 조용히 처리되므로, 사용자는 빠르게 피드백을 받을 수 있음

  3. 실제 서비스 맥락에서의 예시 (소셜미디어 좋아요/팔로우/리트윗)

     | 상황        | 일반적 방식                                                                      | 낙관적 업데이트                                          |
     | ----------- | -------------------------------------------------------------------------------- | -------------------------------------------------------- |
     | 액션        | 좋아요 누름                                                                      | 좋아요 누름                                              |
     | 사용자 경험 | 버튼이 잠시 비활성화 되거나 로딩 표시 → 0.5초 후 숫자가 1 증가하고 버튼색이 바뀜 | 버튼을 누르는 즉시 숫자가 1 증가하고 버튼 색이 바뀜      |
     | 효과        |                                                                                  | 느리다는 느낌을 해소하고 사용자에게 즉각적인 만족감을 줌 |
- TanStack Query 기반 구현 흐름을 `onMutate → (mutate) → onError → onSettled` 순서로 기술해주세요..
  | 단계        | 실행 시점                         | 주요 역할                                      | 결과                               |
  | ----------- | --------------------------------- | ---------------------------------------------- | ---------------------------------- |
  | `onMutate`  | 서버 요청 직전                    | 낙관적 UI 업데이트 및 롤백 데이터 저장         | 캐시가 임시로 변경됨               |
  | `(mutate)`  | onMutate 성공 후                  | 실제 서버 데이터 요청                          | 서버 응답 (성공/실패)              |
  | `onError`   | 서버 요청 실패 시                 | 롤백: 캐시 데이터를 onMutate의 스냅샷으로 복구 | 캐시가 이전 상태로 돌아감          |
  | `onSettled` | 서버 요청 종료 시(성공/실패 무관) | 최종 동기화: 관련 쿼리 무효화 및 Refetch       | 캐시가 서버의 최신 데이터로 확정됨 |
- ToDo “생성” 및 “좋아요 토글”에 **낙관적 업데이트(OptimisticUpdate)**를 적용했을 때의 **실패/충돌 롤백 전략**을 설계해주세요. 1. ToDo ‘생성’ 액션의 롤백 전략

          ToDo 목록에 새 항목을 추가하는 액션은 주로 “생성된 항목을 목록에서 제거”하는 방식으로  롤백됨

          ⇒새 ToDo 항목을 목록에 즉시 추가하고, 실패 시 이 임시 항목을 즉시 제거

          | 단계 | 역할 | 구현내용 |
          | --- | --- | --- |
          | onMutate | 낙관적 적용 & 스냅샷 저장 | 1. 쿼리 취소: queryClient.cancelQueries([’todos’])를 호출하여 진행중인 todos 쿼리를 취소한다. 2. 낙관적 추가: 클라이언트 전용 임시 ID를 가진 새 ToDo 객체를 생성하여 기존 todos 목록의 맨 앞/뒤에 즉시 추가 3. 원본저장: 변경 전의 todos 목록 전체를 context 객체에 저장하고 반환 |
          | (mutate) | 서버 요청 | 서버 API(POST/todos)를 호출하여 새 ToDo 생성(임시 ID는 제외하고 전송) |
          | onError | 실패 롤백 | 1. 원본 복구: `onMutate`에서 저장했던 원본 todos 목록(`context.previousTodos`)을 사용하여 캐시를 덮어 씀.(`queryClient.setQueryData([’todos’],previousTodos`)) 2. 사용자 피드백: 토스트 메시지 등으로 ‘ToDo 생성에 실패했습니다’라고 사용자에게 알림 |
          | onSettled | 최종 일관성 확보 | 쿼리 무효화: `queryClient.invalidateQueries({queryKey:[’todos’]})`를 호출하여 todos 목록 쿼리를 무효화하고 Refetch를 강제함 |
      2. ToDo ‘좋아요 토글’ 액션의 롤백 전략

          이미 존재하는 ToDo 항목의 좋아요 상태를 변경하는 액션은 주로 “변경 이전의 상태 값으로 복원”하는 방식으로 롤백됨

          ⇒ 좋아요 상태와 횟수를 즉시 변경하고, 실패 시 이 상태를 정확히 이전 값으로 되돌림

          | 단계 | 역할 | 구현내용 |

  | --- | --- | --- |
  | onMutate | 낙관적 적용 & 스냅샷 저장 | 1. 쿼리 취소: `queryClient.cancelQueries(['todos'])` 및 해당 ToDo의 상세 쿼리(`['todo', todoId]`)를 취소합니다. 2. 원본 저장: 변경하려는 특정 ToDo 항목의 원본 객체를 `context.originalTodo`에 저장하고 반환합니다. 3. 낙관적 업데이트: 캐시 내에서 해당 ToDo 항목을 찾아 `isLiked` 상태를 뒤집고, `likeCount`를 `+1` 또는 `1`로 즉시 변경합니다. |
  | (mutate) | 서버 요청 | 서버 API(`PUT /todos/{id}/like`)를 호출하여 좋아요 상태를 토글 |
  | onError | 실패 롤백 | 1. 원본 복구: `onMutate`에서 저장했던 원본 ToDo 객체 (`context.originalTodo`)를 사용하여 캐시 내의 해당 항목을 이전 상태로 덮어씁니다. (isLiked, likeCount 모두 복원) 2. 사용자 피드백: '좋아요 처리에 실패했습니다'라고 사용자에게 알립니다. |
  | onSettled | 최종 일관성 확보 | 1. 쿼리 무효화: `queryClient.invalidateQueries({ queryKey: ['todos'] })`를 호출하여 todos 목록 쿼리를 무효화합니다. 2. 상세 쿼리 무효화: `queryClient.invalidateQueries({ queryKey: ['todo', todoId] })`를 호출하여 개별 ToDo 상세 쿼리도 무효화합니다. (이 과정이 충돌 해결의 핵심입니다. 서버가 토글 요청을 성공적으로 처리했더라도, 다른 사용자의 좋아요 변경 사항을 포함한 최신 데이터로 캐시를 최종 덮어쓰게 됩니다.) |
