```Referential Equality```
- 참조 동일성은 "두 변수가 완전히 같은 객체를 가리키고 있는가를 보는 개념이다."

1. 자바스크립트에서의 값 vs 참조

- primitive 타입
    number,string,boolean,null,undefined,bignit,symbol 등 변수 안에 값 자체가 들어있는 경우.

2. object 타입
변수 안에 "객체가 저장된 메모리 위치"가 들어있는 경우.

== (느슨한 동등)
- 타입을 암묵적으로 변환하면서 비교.
- 예측하기 어려운 경우가 많아서 실무에서는 거의 안씀.

===(엄격한 동등)
- 타입 변환 없이 그대로 비교
- primitive에서는 값 동일성
- 객체/배열/함수에서는 "참조 동일성"을 비교한다.

Object.is
기본적으로 ===와 비슷하지만,
Nan과 비교, +0,-0 같은 특수케이스에서 조금 다르다.

객체에 대해서는 ===와 똑같이 참조 동일성을 비교한다.

4. 참조 동일성이 중요한 이유

1. 가변 객체 공유로 인한 사이드 이펙트

2. 불변 객체 + 참조 동일성
- 원본은 그대로 두고 객체를 만들어서 반환.
    - 과서 상태를 안전하게 보관할 수 있고, 참조가 바뀌었는지만 봐도 변경 여부를 어느 정도 알 수 있다.

3. 상태를 직접 수정하면서, 참조 비교로 변경 여부를 보려는 경우.

- **리액트 렌더링 최적화**와 **`Referential Equality`**는 어떤 관계가 있을까요? 
    
 -    렌더링 최적화는 대부분 “참조 동일성(===)” 으로 변경 여부를 판단하기 떄문에, Refrential Equality가 깨지면 최적화가 바로 무력화 된다.

 # **`useCallabck`** 이 무엇인지?

---

- 함수(콜백)를 “메모이제이션” 한다는 게 무슨 뜻인지?
    - 렌더링이 반복되어도 같은 함수 참조를 재사용한다는 뜻이다
    - React에서 함수는 객체이고, 랜더링  시 기본적으로 새 함수를 생성한다.
    - useCallback은 이전에 만들어졌던 함수를 기억해 재사용 하는 것이다.
- 언제 새 함수를 만들고, 언제 기존 함수를 재사용하는지?
    - deps 배열이 바뀌면 새 함수를 생성하고
    - deps 배열이 바뀌지 않으면 → 기존함수를 재사용한다.

# 왜 **`useCallabck`**을 사용하는지?

---

- **불필요한 리렌더링 방지**와 어떤 관련이 있는지
    
    React는 props 와 비교시 참조(===)로 판단한다
    
    함수가 새로 만들어지면 자식은 props 변경으로 인식함
    
    useCallback→함수 참조 고정 → 자식 리렌더링을 방지하는 과정
    
- 성능 최적화 관점에서 얻는 이득 vs 남용했을 때의 오버헤드
    
    이득은 자식 컴포넌트의 리렌더링 감소 및 useEffect 반복실행을 방지한다.
    
    오버헤드는 코드복잡도가 증가하고 메모이제이션 비용의 문제로 인해 
    
    → 리렌더링 비용이 클때만 사용한다.

# **`useCallabck`** 기본 사용법

---

- **`useCallabck`**은 어떻게 사용하나요? (코드)
    
    ```tsx
    const handleClick = useCallback(() => 
    	setCount(prev=>prev+1);
    	}.[]);
    ```
    
- `deps` 배열에 무엇을 넣어야 하는지 규칙
    
    콜백 내부에서 사용하는 외부값은 deps에 포함한다.
    
    외부 값으로는 state, props, 변수 등
    
- 의존성 변경 시 콜백이 어떻게 다시 만들어지는지
    
    deps가 같다 → 함수 재사용
    
    deps를 변경해야한다 → 새 함수를 생성한다.

# **`useCallabck`**에서 중요한 개념

---

- **참조 동일성(reference equality)** 이 왜 중요한지 (=== 비교)
    
    React는 함수의 내용이 아니라 참조만을 비교한다. 따라서 useCallback은 참조를 유지하는 역할이다.
    
- 클로저와 상태: 콜백 안에서 state, props를 사용할 때 주의할 점
    
    ❌
    
    ```tsx
    useCallback(() => {
    setCount(count +1);
    }, []);
    
    ```
    
    - 오래된 `count` 사용
    
    ✅
    
    ```tsx
    useCallback(() => {
    setCount(prev => prev +1);
    }, []);
    
    ```
    
- **stale closure(낡은 값 캡처)** 문
    - deps에 필요한 값을 넣지 않았을 때
    - 콜백이 **옛 state/props를 기억**
    
    **해결**
    
    - deps 정확히 작성
    - 함수형 업데이트 사용
- **stale closure(낡은 값 캡처)** 문제는 언제 생기는지, 어떻게 피하는지
    
    deps에 필요한 값을 넣지 않거나 콜백이 전의 state/props를 기억할때 발생하고
    
    deps를 정확히 작성하거나 함수형 업데이트를 사용하는 방식으로 해결한다.

    # 이벤트 핸들러 / 비동기 로직에서 **`useCallabck`** 예시

---

- 버튼 클릭 시 API 호출하는 핸들러를 `useCallback`으로 감싸는 패턴
    
    ```tsx
    const onClickFetch = useCallback(async () => {
      await fetch("/api/todos");
    }, []);
    
    ```
    
- `useEffect` 안에서 의존성으로 콜백을 넣을 때 패턴
    
    ```tsx
    const load = useCallback(async () => {
      await fetch(`/api/users/${userId}`);
    }, [userId]);
    
    useEffect(() => {
      load();
    }, [load]);
    
    ```
    
- 폼 제출 핸들러, 디바운스/스로틀 함수와 함께 사용할 때

# **`memo`**에 대하여 정리해주세요!🍠

---

- **`memo`**가 무엇인지? 🍠
    
    # **`memo`**가 무엇인지?
    
    ---
    
    컴포넌트를 메모이제이션하여 props가 바뀌지 않으면 리렌더링을 건너뛰게 해주는 기능이다.
    
    즉 같은 입력 → 같은 출력이면 굳이 다시 그리지 않게 하는 것
    
- 왜 **`memo`**를 사용하는지? 🍠
    
    # 왜 **`memo`**를 사용하는지?
    
    ---
    
    리엑트는 기본적으로 부모가 리렌더되면 자식도 같이 리렌더될 수 있음. 그러나 자식이 무겁거나 많거나, 자주 렌더 되는 구조이면 성능이 떨어질 수 있기 떄문
    
    이떄 memo를 사용하면 
    
    - 불필요한 자식 리렌더를 줄여서 성능 개선
    - 리스트 아이템/카드/행 같은 반복 UI에서 효과가 크다.
- **`memo`** 기본 사용법 🍠
    
    # **`memo`** 기본 사용법
    
    ---
    
    ```tsx
    import React from "react";
    
    type Props = { title: string };
    
    const Card = React.memo(function Card({ title }: Props) {
      console.log("Card render");
      return <div>{title}</div>;
    });
    
    export default Card;
    ```
    
- **`memo`**를 언제 쓰면 좋은지 / 안 좋은지 🍠
    
    # **`memo`**를 언제 쓰면 좋은지 / 안 좋은지
    
    ---
    
    ## ✅ 쓰면 좋은 경우
    
    - **리스트 아이템**(TodoItem, Card, Row 등)처럼 개수가 많고 자주 갱신됨
    - 자식 컴포넌트가 **렌더 비용이 큰 경우**(복잡한 UI/차트/지도)
    - props가 **자주 안 바뀌는데** 부모 렌더가 잦은 구조
    - “이 컴포넌트는 props만 같으면 항상 똑같이 보인다”가 확실할 때
    
    ## ❌ 안 좋은 경우
    
    - 컴포넌트가 가볍고 단순해서 **그냥 렌더하는 게 더 싼 경우**
    - props로 내려주는 값이 매번 새로 만들어짐
        
        (예: `onClick={() => ...}` / `{}` / `[]` 같은 새 객체/함수)
        
        → 이러면 `memo`여도 **props가 바뀐 걸로 판단**해서 다시 렌더됨

---

- **`useMemo`**가 무엇인지? 🍠
    
    ---
    
    `useMemo`는 **“계산 결과(값)를 메모이제이션”** 하는 훅이다.
    
    렌더링 중에 어떤 값을 계산할 때, **의존성(deps)이 바뀌지 않으면 이전에 계산했던 값을 그대로 재사용함.**
    
    - `useCallback` = 함수(콜백) 메모
    - `useMemo` = 값(계산 결과) 메모
- 왜 **`useMemo`**를 사용하는지? 🍠
    
    
    ---
    
    - **비싼 계산을 매 렌더마다 다시 안 하려고 할때.**
        
        (정렬/필터/대규모 데이터 가공, 복잡한 수식 등)
        
    - **참조 동일성(referential equality)을 유지하려고 할 때**
        
        객체/배열을 매번 새로 만들면 props가 “바뀐 것”으로 인식되서
        
        `memo`된 자식이 **불필요하게 리렌더**될 수 있음 → `useMemo`로 주소를 고정
        
- **`useMemo`** 기본 사용법 🍠
    
    # **`useMemo`** 기본 사용법 🍠
    
    ---
    
    ```tsx
    
    import { useMemo } from "react";
    
    const sorted = useMemo(() => {
      return items.slice().sort((a, b) => a.name.localeCompare(b.name));
    }, [items]);
    ```
    
- **`useMemo`**에서 중요한 개념 🍠
    
    ---
    
    ### 1) 의존성 배열(deps)이 핵심
    
    `useMemo(() => 값, [deps])`에서
    
    **deps에 들어간 값이 바뀌면 재계산**, 안 바뀌면 재사용.
    
    ### 2) “성능 최적화 힌트”이지 “보장”은 아님
    
    리액트는 상황에 따라 메모를 버릴 수도 있다.
    
    (그래서 `useMemo` 결과에 **의존해 로직이 깨지면 안 됨**)
    
    ### 3) “값 저장” 용도가 아님
    
    - `useMemo`는 캐시(재계산 방지) 목적
    - “상태처럼 유지해야 하는 값”은 `useState` / `useRef`가 맞을 수 있음.
    
    ### 4) 남용하면 오히려 손해
    
    `useMemo` 자체도 비용이 있어서, 가벼운 계산이면 그냥 매번 계산하는 게 더 나을 때 많다.
