gcTime과 staleTime의 차이점에 대해 정리해보세요 🍠

- `gcTime`은 무엇인가요? 🍠

  데이터가 캐시에서 **사용되지 않은 상태로 얼마나 오래 남아 있을지** 결정하는 시간

  - 상태 기준: 데이터가 더이상 어떤 컴포넌트에서도 사용되지 않는 시점부터 타이머가 작동한다
  - 브라우저 동작: gcTime이 만료되면, React Query는 해당 데이터를 **캐시에서 완전히 제거**함
  - 목적: 사용되지 않는 데이터가 메모리를 차지하는 것을 방지하여 **메모리 효율을 관리**하기 위함
  - 기본값: 5분(5*60*1000ms)

  핵심: **메모리 해제**

  gcTime이 만료되면, 데이터를 다시 사용하려고 할 때 캐시에 없으므로 **처음부터 다시 서버 요청**을 보내게 됨

- `staleTime`은 무엇인가요? 🍠

  데이터가 **‘신선한(Fresh)’ 상태로 유지되는 시간**

  - 상태 기준: 데이터가 이 시간 동안은 “**Fresh**” 상태로 간주됨
  - 브라우저 동작: Fresh 상태인 동안에는 컴포넌트가 마운트되거나, 윈도우 포커스가 돌아와도 **새로운 데이터를 서버에서 가져오지 않음**. 캐시된 데이터를 즉시 사용함.
  - 목적: 불필요한 네트워크 요청을 방지하여 **성능을 최적화**하고 사용자에게 **빠르게 화면을 표시**하기 위함
  - 기본값: 0 밀리초(즉, 데이터가 캐시에 들어가는 순간부터 바로 ‘Stale(오래된)’ 상태가 됨)

  핵심: **재요청 방지**

  staleTime이 남아있는 동안에는 React Query는 데이터를 최신 상태로 간주하고, 데이터를 사용하는 훅(useQuery)이 다시 실행되더라도 **서버에 요청을 보내지 않음**

- 두 값을 어떤 식으로 설정하여야 `캐싱 전략에 유리`한가요? 🍠
  핵심: **사용자 경험을 해치지 않으면서도** 불필요한 요청을 줄이는 것
  ### A. staleTime 설정 전략(성능 최적화)
  staleTime은 **데이터의 변경 빈도**에 따라 다르게 설정해야 유리함
  1. 자주 변하지 않는 데이터(`staleTime:Infinity`):
     - 예시: 사용자 설정, 국가 목록, 웹사이트 약관 등
     - 설정: Infinity. 수동으로 데이터를 업데이트 하기 전까지는 절대 재요청을 보내지 않게 하여 네트워크 요청을 완전히 제거한다
  2. 가끔 변하는 데이터 (`staleTime: 5분~1시간`):
     - 예시: 게시판 목록, 상품 상세 정보 등
     - 설정: 사용자가 같은 페이지를 짧은 시간 내에 여러 번 방문하더라도 요청을 한 번만 보내도록 넉넉하게 설정
  3. 자주 변하는 데이터(`staleTime:0`):
     - 예시: 실시간 주가, 알림 목록, 채팅 메시지 등
     - 설정: 0 또는 낮은 값. 사용자가 페이지를 볼 때마다 최신 데이터를 가져오도록 하여 데이터의 신뢰성을 높임
  B. gcTime 설정 전략(메모리 관리)
  gcTime은 일반적으로 기본값(5분)을 유지하는 것이 가장 좋음
  1. 기본값 유지(gcTime: 5분):
     - 대부분의 애플리케이션에서는 사용자가 5분 이내에 해당 데이터를 다시 사용하는 경우가 많으므로, 5분 동안 캐시를 유지하는 것이 다시 요청을 보내는 것보다 효율적
  2. 데이터가 매우 큰 경우(gcTime을 낮춤):
     - 캐시된 데이터의 용량이 매우 커서 메모리 사용량이 부담된다면, gcTime을 1~2분 정도로 짧게 설정하여 사용하지 않는 데이터를 더 빨리 메모리에서 해제
  3. 매우 자주 사용되는 경우(gcTime: 10분 이상):
     - 앱의 핵심 데이터가 자주 사용되지만 메모리에 부담이 없다면, gcTime을 길게 설정하여 캐시가 유지되도록 함. (예: 메뉴바에 항상 필요한 사용자 프로필 데이터)
  결론적으로, `staleTime`을 조정하여 성능(요청 횟수)을 조절하고, `gcTime`은 대부분의 경우 기본값을 유지하면서 메모리 효율에 초점을 맞추는 것이 가장 유리한 캐싱 전략이다.
